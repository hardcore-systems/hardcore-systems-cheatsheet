# Seastar Shard-per-Core 极速无锁网络框架速查卡片

## M1: Shard-per-core 极速无锁调度架构

### Card 1: Shard-per-Core 核心无共享架构
- **核心机制**：Seastar 抛弃了传统的动态多线程调度，全面采用 Shard-per-Core（单核物理绑定）架构。每一个 CPU 物理核心上仅运行一个专属的 OS 线程，该线程拥有一个独立的事件循环（Reactor）及全部资源。
- **无锁化优势**：核心与核心之间完全隔离，不存在任何数据共享（Shared-Nothing）。这消除了锁机制（Mutexes）与原子竞争，让 CPU 效率随物理核数线性扩展。
- **物理折衷**：由于完全不共享，若需跨核读写数据，必须以消息发送的形式进行，增加了微量的内存开销。

### Card 2: CPU 亲和性绑定与核心独占
- **硬件硬绑定**：Seastar 在启动时，将各个工作线程通过系统调用（如 `sched_setaffinity`）强制绑定（Pinning）到特定的 CPU 物理核心。
- **零中断抖动**：通过隔离 CPU，该核心不再参与 OS 的普通进程调度，杜绝了物理上下文切换（Context Switch）和 CPU L1/L2 缓存行被污染。
- **防范建议**：宿主机必须配置 `isolcpus` 引导选项，强行将 Seastar 的核心从系统任务分配中隔离出来，确保真正的独占。

### Card 3: 无锁物理内存分配器
- **内存本地化**：传统 `malloc` 在多线程分配时会因为全局堆锁（Global Heap Lock）产生严重挂起。Seastar 自定义无锁分配器，在初始化时静态瓜分物理内存，每个 CPU 核心分得专属的 Page 区域。
- **物理分配**：本核的内存请求完全由本核内存段内 $O(1)$ 级分配，规避了一切跨核内存锁抢占。
- **生命期限制**：如果在一个核心上分配的内存被发送并由另一个核心释放（Cross-shard Free），Seastar 会通过无锁环形队列将其退回给原始分配核执行异步释放。

### Card 4: 协作式协程调度与纤维
- **协作调度**：Seastar 的工作线程内部使用协作式多任务（Cooperative Multi-tasking）。同一个 CPU 核心内运行着大量轻量级纤维（Fibers）。
- **非抢占限制**：Fibers 不能被操作系统抢占。只有当当前 Fiber 遭遇异步等待（如 I/O、Future）并主动出让（Yield）控制权时，核心 Reactor 才会切换至下一个任务。
- **系统限制**：如果用户代码中写了长耗时同步阻塞循环，会直接卡死该核心的 Reactor 调度，导致网络心跳超时，需要极力防范。

### Card 5: 跨核心无锁 SPSC 环形队列
- **跨核通信**：当 Core A 必须通知 Core B 运行某项任务时，它不直接访问 Core B 内存，而是将任务封装成一个 Message 投入到 SMP 专属的无锁单生产者单消费者（SPSC）环形队列中。
- **无锁原子更新**：通过原子的读写指针偏移控制，写入和读取在不同的缓存行（Cache Lines）完成，零锁竞争。
- **折中优化**：环形队列有固定的容量限制。如果队列写满（Queue Overflow），Core A 将触发背压，暂停前台事件接收并等待 Core B 消费。

## M2: 异步非阻塞编程模型

### Card 6: Future 与 Promise 核心机制
- **状态容器**：
  - **Future**：表示一个尚未完成但未来可期的异步计算结果。
  - **Promise**：代表异步计算的生产者，用于在计算完成时填充值或异常。
- **非阻塞执行**：Seastar 的 `future` 物理设计极其精简，不含任何互斥锁，专门针对 Reactor 级单线程环境进行了深度优化。
- **折中折衷**：由于消除了同步锁保护，一个 Future 实例只能由其绑定的 Shard 进行读写，跨 Shard 直接调用会发生数据竞争崩溃。

### Card 7: 链式 Continuation 优化
- **链式组合**：通过 `.then()` 算子，可以将多个异步操作级联起来，构建复杂的执行管线（如 `ReadPacket().then(Parse).then(WriteBack)`）。
- **内存优化**：Seastar 优化了链式调用中的内存布局，利用移动语义（Move Semantics）直接将前置 Future 的结果转移给后续的 Continuation，消除了多余的数据拷贝。
- **诊断难度**：高频的链式注册会在运行期打碎控制流，使得传统堆栈分析（Backtrace）失效，只剩下 Reactor 回路。

### Card 8: C++20 原生协程与 co_await 集成
- **协程转换**：Seastar 原生支持 C++20 协程规范。允许用户使用 `co_await seastar_future` 代替繁琐的 `.then()` 链式写法。
- **编译器编译**：编译器自动将 `co_await` 点以后的代码编译为 Continuation 闭包挂载到 Future 状态机中，让异步代码具备同步编码的可读性。
- **运行损耗**：C++20 协程会产生微量隐式堆内存分配（Coroutine Frame），在追求亚微秒延迟的极限场景下需要权衡使用。

### Card 9: 微秒级任务调度队列
- **队列管理**：Reactor 线程内部维护多个不同优先级的 FIFO 任务队列（Task Queues）。
- **运行轮询**：事件循环在每次 Tick 轮询中依次执行就绪的任务。支持配置任务的 CPU 分配权重，确保控制面任务和高吞吐数据面任务的物理隔离调度。
- **性能优势**：任务切换完全是简单的用户态指针跳转，耗时通常控制在几纳秒，远低于操作系统的线程调度开销。

### Card 10: 异常传播与异步资源管理
- **异常捕获**：在 Future 链中，可以通过 `.handle_exception()` 或在协程中使用 `try/catch` 结构捕获来自 Promise 的异常。
- **资源自愈**：利用 `.finally()` 算子，保证无论异步操作成功还是失败，均能安全释放临时分配的物理资源（如文件描述符、网络缓冲区）。
- **折中限制**：异步异常必须沿 Future 链逐级传递，如果有一级链断开或漏配了异常处理器，会导致整个事件的执行线中断并积压脏资源。

## M3: 高性能网络与磁盘 I/O 引擎

### Card 11: Linux io_uring 引擎与异步 Epoll
- **内核零拷贝**：Seastar 在现代 Linux 内核上全面切换至 `io_uring`。它将系统调用（Syscalls）转化为用户态与内核态共享的 Submission Queue（提交队列）和 Completion Queue（完成队列）双环形缓冲区。
- **零切换开销**：通过 io_uring，发送网络包或读写磁盘只需要将 Task 写入队列并触发内核轮询，完全消除了 `epoll_wait` 或 `write` 带来的内核态-用户态特权级切换开销。
- **降级兼容**：在老旧内核上，系统会自动平滑降级至 Epoll 网络驱动。

### Card 12: 零拷贝 Packet 数据包管理
- **流式结构**：Seastar 的 `packet` 采用分段（Fragmented）内存布局。
- **引用计数包**：当一个大网络包需要分发到多个目的地或添加协议头时，Sled 仅生成引用计数的 Fragment 视图，不涉及任何物理数据的拷贝。
- **设计折衷**：非连续内存的 Fragment 会在物理网卡发送时增加 Scatter-Gather I/O 链的构建开销，但相比大内存拷贝其带宽优势依旧极其巨大。

### Card 13: 物理 DMA 与 O_DIRECT 磁盘 I/O
- **Direct I/O**：磁盘读写通过配置 `O_DIRECT` 强制绕过 Linux 内核 Page Cache。
- **零内存抖动**：数据通过物理 DMA 直接在磁盘控制器与 Seastar 用户态内存之间传输。这消除了操作系统 Page Cache 带来的全局锁竞争和内存挤压。
- **开发折衷**：由于失去了 OS 缓存，开发者必须在应用层显式构建并管理自己的 Block Cache，增加了数据库和文件系统的业务层实现成本。

### Card 14: 磁盘 I/O 调度器与公平队列
- **多核心公平**：在 Shard-per-Core 下，数十个核心可能同时发起 DMA 磁盘请求，从而挤爆物理 SSD 控制器队列，导致严重的 I/O 饥饿。
- **I/O Fair Queue**：Seastar 实现了核心级的 I/O 调度器。每个核心拥有一个发包配额令牌（Token）。调度器根据物理硬件的 IOPS / 带宽瓶颈，在多核间以公平队列算法配额派发，确保没有单个 Core 能垄断磁盘带宽。
- **参数调优**：需要在启动前利用 `seastar-io-setup` 工具自动测出磁盘的实际物理参数上限并写入配置。

### Card 15: DPDK 用户态极速协议栈
- **内核旁路（Kernel Bypass）**：Seastar 支持集成 DPDK。直接劫持物理网卡，在用户态接管全部网络收发包处理。
- **零拷贝协议**：绕过整个 Linux 内核 TCP/IP 协议栈，直接在用户态完成 ARP/IP/UDP/TCP 等各层协议的手写解析，数据包处理延迟直接压降至微秒级。
- **硬件局限**：DPDK 强行霸占网卡，要求该网卡必须由专用驱动托管，普通 OS 网络工具（如 `ifconfig`、`iptables`）将彻底失效。

## M4: 无锁并发与多核拓扑

### Card 16: 原子无锁共享结构
- **单核屏障**：由于每个 Core 都拥有独立的 Reactor，如果必须要维护一些全局只读/稀疏修改的全局状态（如全局节点配置），Seastar 采用原子变量（`std::atomic`）配合 Memory Barriers 来保证跨核可见性。
- **控制冲突**：尽量限制在只读读取场景，任何高频的跨核原子写都会强行导致多核间的 CPU 缓存一致性总线（Cache Coherency Bus）风暴，拖慢 Reactor 运行。

### Card 17: Read-Copy-Update (RCU) 读写分离机制
- **零锁读保障**：对于需要动态更新的高频共享数据，Seastar 鼓励采用类 RCU 思想。
- **原子替换**：读取线程通过指针读取旧版本的只读快照数据，完全无锁；写入线程复制数据副本，在副本上完成修改，并原子交换指针，随后利用 EBR（Epoch Based Reclamation）等跨核延迟释放机制异步销毁旧版本。
- **空间消耗**：以多版本占用少量物理堆空间换取查询路径上的零锁和零原子操作开销。

### Card 18: 多核并发屏障与协同同步
- **初始化屏障**：在系统启动（App Startup）与退役（App Shutdown）阶段，所有 Shards 需要协同同步，确保全部 Reactor 已准备就绪或安全退火。
- **异步 Barrier**：Seastar 提供非阻塞的 `smp::invoke_on_all` 屏障接口，返回一个包含所有 Shards 状态的 Future 数组，在完成同步时触发回调。
- **运行策略**：绝不在运行期使用阻塞的 `std::barrier` 或 CPU 自旋锁，防止直接击穿 Shard-per-Core 协作调度的低延迟保护。

### Card 19: 缓存行对齐对冲伪共享
- **物理现象**：当两个工作核心（Core A & Core B）在不加锁的情况下高频读写在物理内存中相邻（落在同一个 64 字节 Cache Line）的变量时，由于 CPU 硬件一致性总线的限制，该缓存行会在多核间频繁失效，导致写入性能骤降十倍。
- **Cache Line Padding**：Seastar 在关键的跨核通信结构和核心私有变量上，强制使用 `alignas(64)` 或 padding 字节做缓存行物理对齐。
- **折中折衷**：虽然会浪费微量的内存字节空隙，但换取了极度重要的多核并发并行度。

### Card 20: CPU 拓扑与 NUMA 感知内存
- **亲和内存**：在多 CPU 插槽（NUMA 架构）服务器中，CPU 核心访问本地插槽绑定的物理内存条速度极快，而跨插槽访问（Cross-socket）则会产生数倍的延迟。
- **NUMA Local Allocation**：Seastar 的内存分配器感知宿主机物理拓扑，确保 Core $N$ 线程申请的内存百分之百位于其所绑定的 CPU 插槽的 NUMA 本地节点上。
- **部署建议**：启动时可配置 `--numa-node` 参数指定分配范围，配合 NUMA 控制工具绑定。

## M5: 分布式应用支撑与微服务

### Card 21: 跨 Shard 分布式服务协同
- **分布式服务**：Seastar 提供了 `sharded<Service>` 模板类。它会在所有激活的 CPU 核心上各自实例化一份 Service 实例。
- **本地化查询**：当客户端请求到达时，系统根据负载或连接归属，直接将其路由到当前核心的本地 Service 实例中进行计算，绝无跨核锁冲突。
- **数据同步**：在需要收集全局状态时，利用 `sharded::map_reduce` 将各个核心的服务状态以无锁消息形式汇总。

### Card 22: 连接管理器与连接速率限制
- **自适应过载保护**：每个 Shard 内置 Connection Manager，负责跟踪当前 Reactor 的活跃网络连接。
- **水位限制**：当单核连接数超过硬上限，或者 Reactor CPU 负荷触及限流区间时，主 Reactor 会主动执行拒绝策略（如丢弃或重定向连接请求）。
- **设计折衷**：保护了已建立连接的高质量响应，避免整个系统在突发流量攻击下出现雪崩。

### Card 23: 背压流控自愈
- **队列自愈**：在跨核心或网络异步发包过程中，由于发送和消费速率的不对称，临时缓冲区会堆积。
- **背压传导**：当 SMP 跨核队列或 TCP 发送缓冲区到达特定阈值时，Seastar 的 Flow Control 机制会自动减慢或暂停读取上游 Socket，逼迫客户端放慢发包速率，实现了自底向上的物理限流。
- **折中优化**：这可能会使部分客户端观察到延迟短暂上升，但保护了内存容量，防范因缓冲区无限膨胀导致物理 OOM 强退。

### Card 24: 零拷贝极速 HTTP/JSON 解析
- **极致网络效率**：Seastar 内置了高性能的 HTTP 协议解析器和 JSON 编译器。
- **直接读段**：解析器直接在网络 Packet 缓存片段（Fragments）上进行原位字符串切片与跳转，无需将数据复制到应用层字符串变量中。
- **适用场景**：适合构建云原生高性能微服务 API 网关，单核可以轻松吃满万兆网卡带宽极限。

## M6: 系统运行参数调优与诊断

### Card 25: CPU 挂起检测与慢任务监控
- **Stalled Reactor Detection**：Reactor 线程内部运行着一个微秒级的实时看门狗时钟。
- **超时报警**：如果某个用户任务（Continuation / Coroutine）霸占 CPU 连续运行时间超过预设阈值（默认 500 微秒），看门狗会记录一条 Stalled Thread 警告，并在调试模式下打印出当前阻塞处的堆栈信息。
- **诊断用途**：是排除前台代码中误写了系统级阻塞 Syscall（如 `std::this_thread::sleep_for`）的首选遥测监控手段。

### Card 26: 异步堆栈回溯与调试诊断
- **堆栈重建**：由于异步 Future 的控制流由 Reactor 轮询切碎，传统的 `gdb` 回溯只能看到 `reactor::run()` 这一行。
- **解决方案**：Seastar 提供了异步调试辅助工具和特殊的运行时 Span 链，在 Debug 状态下能够逻辑还原当前 Task 之前的 Future 依赖链条，拼装出真实的异步调用树。
- **折中开销**：跟踪异步上下文会引入一定的内存分发开销，建议只在 Debug 和测试环境中开启，生产环境编译时务必关闭。

### Card 27: 运行期核心监控指标遥测
- **丰富探针**：Reactor 暴露了数十个 Prometheus 监控遥测指标：已完成的任务数（tasks_processed）、当前挂起的 Futures（pending_futures）、跨核信息量（cross_shard_requests）、io_uring 提交率。
- **系统调优**：通过实时监控这些指标，可以判断系统是否发生了多核间任务分配不均（负载倾斜）或磁盘 I/O 队列阻塞。

### Card 28: C++ 编译期参数调优字典
- **极致编译器优化**：开发 Seastar 应用程序时，编译选项对最终性能影响极其巨大。
- **调优组合**：
  - `-O3` 开启编译器全优化。
  - `-flto`（链接期优化）可跨目标文件内联 Future 虚调用，将性能提升 15% 以上。
  - `-march=native` 开启针对当前物理 CPU 的 SIMD/缓存行优化编译。
- **注意要点**：启用 LTO 会导致编译耗时大幅延长，物理内存开销倍增，适合做发布编译（Release Build）时配置。
