# Sled 现代嵌入式 B-Tree 存储引擎速查卡片

## M1: 无锁 B-Link Tree 与逻辑结构

### Card 1: 无锁 B-Link 树核心原理
- **共识机制**：Sled 底层基于 B-Link Tree。传统的 B 树进行分裂时，需要锁住整条从根到叶子的路径；B-Link 树通过给每个节点引入右指针（Right-Sibling Link）及边界键值，使得读取线程在遇到正在分裂的节点时，可以直接顺着右链接遍历到新分裂出的兄弟节点，从而彻底免除读锁。
- **并发优势**：读操作从不阻塞写操作，极大地平抑了多核机器上的并发延迟瓶颈。
- **折中限制**：由于没有全局路径锁，树的分裂是分步局部生效的，这就需要异步父节点更新机制。

### Card 2: 页表原子 CAS 更新与 Epoch 回收
- **物理寻址**：Sled 使用一个全局页表（Page Table）进行映射。页表不记录真实的物理磁盘地址，而是将逻辑 Page ID 映射到对应页的 Delta 链表头指针。
- **原子更新**：所有针对页的状态修改，都是通过 CPU 原子比较并交换（Compare-And-Swap, CAS）页表的表项指针来完成的。
- **设计折衷**：原子 CAS 规避了传统页表锁，但在超高频并发写时会带来激进的 CPU 缓存行冲突（Cache Bounce），需要靠 EBR（Epoch 垃圾回收）平抑。

### Card 3: 叶子页逻辑排布与数据检索
- **数据结构**：叶子节点（Leaf Node）逻辑上包含排序好的键值对（Key-Value Slots）。在内存中，它被组织为可以利用二分查找快速检索的紧凑数组。
- **写路径**：写入新数据不是就地修改原页，而是向该页对应的 Delta 链追加一个 Put 类型的节点。
- **读路径**：读取时，首先通过二分查找检索 Base 页，然后沿着 Delta 链向上合并最新改动。

### Card 4: 内部节点路由与并发跳转
- **路由寻址**：内部节点（Inner Nodes）仅存储子节点的 Page ID 以及范围边界键。
- **右链接路由**：如果在跳转子节点的过程中，发现子节点已完成物理分裂但父路由项尚未更新，查询线程能通过右指针动态校正，沿右侧物理跳转至正确的区域。
- **设计折衷**：这保证了遍历不中断，但稍微增加了读路径上的分支判断开销。

### Card 5: 读写并发折衷与无锁优势
- **并发对比**：
  - **有锁 B 树**：读写互相排斥，高并发写会导致全树频繁加锁，吞吐发生滑坡。
  - **Sled B-Link**：利用 CPU Atomics 与 Right-Link，读操作零锁，写操作在单页 CAS 竞争，局部收敛。
- **写放大控制**：借由 Delta 链写入规避了每次微小写都要刷新整页的严重写放大（Write Amplification）。

## M2: 日志结构页存储与 Delta 链

### Card 6: 日志物理追加写与 Flash 闪存友好
- **物理布局**：Sled 采用日志结构（Log-Structured）写入。所有的页创建、修改、删除和合并操作，在物理上被统一成追加日志项，顺序写入磁盘文件的活跃 Segment 中。
- **硬件磨损控制**：顺序写完美契合了 SSD 闪存的垃圾回收（Garbage Collection）机制，规避了大量随机写带来的物理磨损，延长固态盘寿命。
- **设计折衷**：磁盘数据不能就地更新，这会导致物理数据文件空间虚高，必须引入后台 GC 压缩。

### Card 7: 页 Delta 链表物理设计
- **页逻辑表示**：在内存中，一个逻辑页的状态用一个链表（Delta Chain）表示。链表的末端是 Base Page（只读扁平数据段），前面是一个个 Delta 节点（如 `Put(k, v)`、`Delete(k)`）。
- **指针变更**：新写入的 Delta 节点指向旧的链表头，并用原子的 CAS 操作将全局 Page Table 中对应的指针指向新 Delta 节点。
- **性能滑坡**：Delta 链如果拉得太长（如大于 10），会导致读取时合并计算的开销飙升，需要执行页合并。

### Card 8: 物理段物理映射与地址重定向
- **寻址路径**：当读取一个 Page 时，Buffer Cache 首先从 Page Table 取得内存指针。如果部分数据块已被置换回磁盘，则根据 Log 位置重定向去磁盘 Block 进行读取。
- **段偏移量**：日志文件中划分为固定 $8MB$ 大小的段（Segments），数据以绝对磁盘 LSN 的偏移寻址。
- **设计折衷**：由于重定向频繁，若磁盘 I/O 拥堵，读延迟会出现阶跃，需要合理的预读机制。

### Card 9: 页合并压实内存整理
- **Page Consolidation**：当一个页的 Delta 链长度超出阈值（默认 8）时，Sled 的后台工作线程会对其进行 Consolidation。
- **物理过程**：创建一个新的、扁平的 Base Page，把 Delta 链中的 Put/Delete 变更合并应用到该新页上。然后通过原子的 CAS 操作，将 Page Table 的指针切换指向新的扁平 Base Page，随后释放旧 Delta 链。
- **资源开销**：合并操作耗费 CPU 并会产生一次新的追加写，这是典型的以少量空间写换取极速读的折衷。

### Card 10: WAL 刷盘与持久化落盘
- **ACID 保证**：写操作追加到内存的 Log Buffer 中。当事务要求 Durability 时，触发 `flush`，将 Buffer 刷盘。
- **Group Commit**：多线程写入时会自动将刷盘操作合并为一次 I/O（组提交），以减少物理磁盘寻道延迟。
- **参数控制**：可以通过配置设置刷盘频率，在追求极限性能和容忍微量掉电数据丢失之间自由折衷。

## M3: 并发页分裂时序与恢复

### Card 11: 叶子页溢出物理分裂时序
- **物理步骤**：当叶子节点的 Key Slots 大小超过物理限制（如默认 $8KB$）时，触发 Split 操作。
- **两阶段执行**：
  - 第一阶段：分配一个全新的 Page ID，将旧页（Leaf A）的右半部分数据拷贝至新页（Leaf B）中，并使 Leaf B 的右指针指向 Leaf A 原先的右兄弟，Leaf A 的右指针则原子重指向 Leaf B。
  - 第二阶段：将分界 Key 异步追加至父节点，完成父路由指向。
- **并发可见**：在一二阶段之间，Reader 能够通过 Leaf A 的右链接无锁读到 Leaf B 的数据。

### Card 12: 分裂期无锁遍历读写路径
- **跳转保障**：在 Split 第一阶段完成后，如果在父节点路由未更新时发起对 B 区 Key 的 Read。
- **遍历轨迹**：遍历线程落入 Leaf A，发现待检索 Key 的范围超出了 Leaf A 的最大界限（Max Key），立即自动顺着 Right-Link 跨步进入 Leaf B，在此读出最新值。
- **设计折衷**：读路径多了一步 `Key > MaxKey` 的逻辑校验，但换取了极度平滑的无锁分裂体验。

### Card 13: 异步父节点路由项更新
- **解耦设计**：为了不阻塞分裂线程，第二阶段的父节点更新（Parent Node Update）是通过一个专门的异步工作线程池完成的。
- **路由级连**：该工作线程向父节点的 Delta 链追加一个 `Split(分界Key, PageB)` 的路由扩展节点。
- **退化防范**：如果父节点更新积压严重，查询线程会面临大量的跨 Right-Link 跳转。Sled 会主动根据跳转深度给前台写入线程施加阻尼限流。

### Card 14: 并发分裂冲突处理与重试
- **冲突场景**：当多个工作线程并发向同一个父节点进行分裂或写入操作时，CAS 页表可能发生失败。
- **自旋重试**：Sled 不使用悲观排他锁，而是采用经典的 Spin-CAS 重试。一旦 CAS 失败（即其他线程抢先更新了父页），当前线程重新读取最新的父页指针，重新合并路由信息，并尝试再次 CAS。
- **极限制约**：如果并发写极度密集，会导致自旋重试次数增加，CPU 功耗上升，需要退回 EBR。

### Card 15: 崩溃恢复与页状态重建
- **冷启动加载**：当 Sled 从意外 Crash 或正常关闭中重启时，必须重新构建全局 Page Table 目录。
- **扫描重构**：从最近的 Checkpoint（检查点）位置开始，顺序向后扫描日志段文件。读取日志中的追加记录（Put/Delete/Split），在内存中重建对应 Page ID 的指针映射与 Delta 链表。
- **物理折衷**：如果运行期长时间没有生成 Checkpoint，重启时的日志扫描耗时会变长。

## M4: LSN 机制与 Epoch 垃圾回收

### Card 16: Log Sequence Number (LSN) 序列号跟踪
- **物理标尺**：Sled 中的每个物理日志实体都关联一个单调递增的 64 位 LSN。
- **排序作用**：LSN 决定了磁盘更新的绝对物理时序。在恢复和 Consolidation 时，Sled 根据 LSN 的大小决定哪个 Delta 节点是更新的版本。
- **设计折衷**：LSN 分配是高频操作，需要内部使用多路原子变量生成器，对性能有极高的要求。

### Card 17: Epoch 内存回收机制 (EBR)
- **并发挑战**：在无锁数据结构中，一个物理 Delta 节点从链表剥离后，不能直接 `free()`，因为此时可能有其他 Reader 线程正沿着链表遍历它，直接释放会导致悬空指针段错误。
- **EBR 原理**：系统维持一个全局 Epoch，所有进入临界区的线程注册当前 Epoch。当节点被替换后，将其置入待回收垃圾队列，仅当所有活跃 Epoch 的线程都退出后，才物理释放。
- **性能优势**：完全消除了全局锁，实现了安全、高性能的并发内存释放。

### Card 18: GC 段清理与磁盘碎片回收
- **Garbage Collection**：日志段中会存在大量的过期 Delta 数据或已被删除的 Page Block（垃圾碎片）。
- **搬迁与回收**：后台 GC 线程监控段的利用率。选中冷 Segment 后，将该 Segment 中仍然活跃的 Page 数据搬迁追加到最新的活跃段中，并将旧 Segment 整体格式化以备循环写入。
- **写放大隐患**：GC 搬迁活跃数据会引发二次物理写入，这是 Sled 必须极力通过水位线控制的写放大源头。

### Card 19: 写放大平抑与磁盘寿命管理
- **SSD 保护**：在 Sled 频繁执行追加写和 GC 时，如果水位线设置不当，会导致闪存物理擦除块（PE Blocks）被高频擦写，磨损 SSD。
- **算法平衡**：Sled 使用了针对闪存调优的垃圾回收启发式算法，延迟数据重写，并对热点/冷数据进行分段堆置。
- **折中优化**：这可能会稍微增加全局文件尺寸（空间占用），但保护了闪存物理介质寿命。

### Card 20: 段归并压实水位线监控
- **阈值监控**：GC 线程通过三个水位线参数（gc_threshold、active_density、stale_ratio）精细控制搬迁时机。
- **自愈响应**：当 stale_ratio（过期数据占比）超过 70% 时，无条件触发 Segment 压实。
- **运维建议**：在高性能嵌入式场景下，建议根据磁盘实际吞吐上限动态微调该水位参数。

## M5: 嵌入式事务与并发 API

### Card 21: 交互式事务 API 与 CAS Loop
- **乐观并发事务**：Sled 提供了完全的 ACID 交互式事务。基于 CAS 乐观循环设计。
- **工作机制**：在事务执行闭包期间，所有的 `insert` 和 `get` 操作并不加锁，而是在本地线程私有缓冲区中记录读写集。在提交时，系统一次性原子 CAS 页表，若中途有其他写操作破坏了读写集一致，事务自动回滚并重新执行闭包。
- **应用折衷**：适合短事务和冲突低的场景；写冲突极高的长事务会导致频繁的 CAS Loop 失败，严重消耗 CPU。

### Card 22: 事务键范围锁管理
- **范围防范**：为了防止幻读（Phantom Read）和非串行化异常，Sled 内部提供了一个轻量级的键范围锁（Key Range Locks）。
- **锁粒度**：只在事务提交评估的短暂时间窗口内锁定受影响的 Key 区间，以防 CAS 重试时读写集发生不可预测的漂移。
- **折中限制**：由于是内嵌式无网络通信，键范围锁完全在内存中维护，开销极低，但不支持跨节点分布式 2PC 协调。

### Card 23: 会话内读己所写一致性
- **Session Consistency**：对于尚未提交的事务，事务内部发起的查询必须能够看到自身之前的写入。
- **物理缓冲**：Sled 在 Thread-Local 事务状态缓冲区中实现缓存：查询时先检索事务私有写缓冲区（Private Write Buffer），若命中则直接返回，若无再下沉读取主页表的 MVCC Epoch。
- **设计折衷**：这稍微增加了事务内读的查找分支开销，但完全实现了单线程内的会话强一致。

### Card 24: 并发客户端线程安全与句柄复用
- **句柄架构**：Sled 的核心 `Db` 对象在 Rust 中实现了 `Send` 和 `Sync`。
- **并发克隆**：客户端只需调用 `db.clone()` 即可将同一数据库句柄分发到几十个不同的物理工作线程中并发操作。
- **锁管理**：底层完全由 EBR 和无锁 B-Link Tree 的页表 CAS 承载，消除了上层的并发通道抢占锁瓶颈。

## M6: 系统运行参数调优与诊断

### Card 25: Block Cache 缓冲区驱逐管理
- **缓存策略**：Buffer Cache 对解密压实后的扁平 Base Page 物理数据进行内存常驻管理。
- **LRU 回收**：基于 Segment 物理位置和最后访问时序的 LRU 算法，在堆内存达到上限时强行物理换出。
- **运行防范**：在运行特大数据科学分析时，建议将 cache 内存上限设为宿主内存的 60%，防范 OS 的 OOM 强杀。

### Card 26: 运行时 Failpoint 混沌注入
- **容错验证**：Sled 的核心物理功能（如页表 CAS、段 GC 搬迁、WAL 刷盘）均埋入了 Failpoint。
- **崩溃注入**：在混沌集成测试中，强行在物理 Split 中途注入 OutOfMemory 或磁盘 IO 损坏异常，验证系统在断电重启后重建 Page Table 的强一致健壮性。
- **设计折衷**：测试代码在 Release 时通过 Rust 条件编译全面剥离，确保不为生产运行带来任何多余的机器指令开销。

### Card 27: 运行期核心监控指标遥测
- **性能诊断**：Sled 暴露了实时的系统运行遥测探针，包括活跃页数（Active Pages）、脏页占比（Dirty Ratio）、EBR 回收挂起队列长度（EBR Pending Queue）、WAL 每秒写入吞吐（WAL IOPS）。
- **接入建议**：在嵌入式环境中，可以将这些指标桥接至后台遥测进程，实时预测存储硬件寿命和性能变差趋势。

### Card 28: 系统运行参数调优字典
- **参数组合**：
  - `segment_cleanup_threshold`：调整 GC 阈值。较大可降低 GC 消耗但增加磁盘占用。
  - `use_compression`：配置列数据是否经过压缩。牺牲 CPU 换取磁盘带宽。
- **终极折衷**：应针对 SSD、NVMe 等不同物理介质，微调 Segment 尺寸（如从 8MB 压降至 1MB），平抑写入放大以延长物联网边缘端闪存寿命。
