# DuckDB 嵌入式分析型 SQL 引擎速查卡片

## M1: 列式存储与内存物理布局

### Card 1: 列式物理布局与行组划分
- **核心机制**：DuckDB 将表的数据按列拆分，并在逻辑上组织为行组（Row Groups），每个行组包含 $122,880$ 行。每一列在当前行组中作为一个连续的数据段（Column Segment）独立存放。
- **物理优势**：行组划分提供水平切分与列式垂直布局的有机结合，极大加速列存扫描。读取只需访问特定列的 Segment，避开了无关列的 I/O。
- **设计折衷**：行组太大限制了删除过滤的细粒度，太小则破坏了连续内存扫描的带宽优势。

### Card 2: 数据块物理分配与段存储管理
- **物理结构**：数据持久化写在单一的大文件（.db）中，底层划分为固定 $256KB$ 大小的数据块（Blocks）。段（Segments）是列数据在 Block 上的映射。
- **元数据索引**：段头记录了数据块的物理偏移量以及数据值的物理编码方式。
- **设计折衷**：对于小表，固定大小的 Block 会产生块内碎片，但大幅简化了空间分配和磁盘回收算法。

### Card 3: 缓冲区管理器与内存限制
- **缓存策略**：Buffer Manager 负责在内存中按需加载和换出数据块。在内存达到用户配置硬上限时，会自动卸载非活跃的列 Block，并将中间状态落入临时溢出文件。
- **数据结构**：基于引用计数和 Eviction Queue（双向链表）管理物理块的状态（PINNED/UNPINNED）。
- **运行防范**：高并发外部扫描中，应避免过长的 PIN 锁块，防止引起锁级死锁或假 OOM。

### Card 4: Zone Map 区间过滤与 Min-Max 索引
- **粗粒度过滤**：每个列段（Column Segment）的元数据中，默认嵌入了 Zone Map。它包含当前段内所有值的最小值（Min）和最大值（Max）。
- **优化路径**：在查询执行前，Filter 算子会直接比对 Zone Map。如果查询过滤条件（例如 `age > 30`）不与该 Zone Map 重叠，直接物理跳过该段，零读盘。
- **系统限制**：如果数据严重无序（如随机乱序），Zone Map 区间会过度膨胀，从而失去过滤能力，必须依赖物理排序。

### Card 5: 内存 WAL 日志与落盘持久化
- **灾难恢复**：对于写事务，DuckDB 使用内存/物理追加写前日志（WAL）。所有对列块的写或更新都会顺序写入 WAL 文件。
- **检查点合并**：执行 Checkpoint 时，WAL 中的变更被批量合并并回写到主文件的各个 Block 中，然后截断 WAL。
- **吞吐折衷**：强 fsync 保证了 ACID，但会拖慢写入 TPS。在纯分析场景下可配置为全内存不持久化以极大提升写速度。

## M2: 向量化计算引擎核心

### Card 6: 向量化计算核心原理
- **批处理加速**：DuckDB 不采用逐行迭代的 Volcano 模型，而是以向量（Vector）为基本计算单元。一个 Vector 通常包含 $1024$ 或 $2048$ 个值，且在物理内存中完全连续分布。
- **内存局部性**：操作在一个紧凑的 `for` 循环中批量遍历这 $1024$ 个元素，这保证了数据能常驻 CPU L1 缓存，规避大量的虚函数调用开销。
- **扩展限制**：向量过大（例如十万）会导致局部数据溢出 CPU L1 缓存，需要根据 L1 缓存大小进行大小调优。

### Card 7: 行式火山模型与向量化执行折中
- **执行模型**：
  - **Volcano**：每次 `next()` 返回一行，包含高频的虚函数调度、分支预测失败和差劲的缓存局部性。
  - **Vectorized**：每次算子调用返回一个 Vector 批数据，在内部展开循环。
- **物理折衷**：向量化在列存分析时能够获得几十倍的 CPU 效率提升；但对于极简单的单行点查，由于需要构建 Vector 容器，其开销反而略高于纯行式引擎。

### Card 8: SIMD 寄存器指令硬件加速
- **并行乘除**：当编译器（如 Clang）感知到向量内部循环的代码模式且无分支时，会自动生成 SIMD（单指令多数据）硬件代码，如 AVX2 或 AVX-512。
- **加速比率**：可以在单个 CPU 周期内并发对 8 个或 16 个 64 位整数进行算术运算或过滤比较。
- **硬件限制**：编译时需要携带特定的指令集选项。如果目标服务器 CPU 不支持，程序启动时会直接崩溃，需要做自适应编译。

### Card 9: 动态类型转换与列投影机制
- **动态类型路径**：在处理表达式（如 `ColA + ColB`）时，Vector 包含其物理类型标志。如果类型不同，DuckDB 会动态生成转换算子（Casting Vector）进行向量级的类型对齐。
- **选择向量（Selection Vector）**：过滤操作不复制数据，而是生成一个索引数组（Selection Vector）指向原向量的合格行，从而实现极速的逻辑裁剪。
- **设计折衷**：选择向量减少了数据拷贝，但会引入非连续物理访存开销，需要后台做规整压实。

### Card 10: 窗口函数向量化缓冲优化
- **状态维护**：窗口函数（如 `LAG`, `LEAD`, `RANK`）在计算时，需要临时记录和排序大量的前后行状态。
- **向量缓冲**：DuckDB 使用专门的 Chunk 缓冲区存储滑动窗口内的列数据，通过向量级位移和指针偏移动态计算窗口结果。
- **内存限制**：在特大窗口下，需要大量缓存，此时系统会自动触发内存上限拦截并启动磁盘写回。

## M3: 事务管理与 Epoch MVCC

### Card 11: 事务管理器与 Epoch 生命周期
- **并发控制**：采用基于 Epoch 的多版本并发控制（MVCC）。系统维护一个单调递增的全局时间戳（Epoch）。
- **事务绑定**：每个事务启动时绑定当前的 Read Epoch，提交时获取新的 System Epoch。
- **设计折衷**：由于是单机内嵌，无需分布式 HLC 时钟，设计极简，但不支持跨物理节点的分布式强一致性横向同步。

### Card 12: MVCC 可见性判别与物理读路径
- **可见性判断**：每行数据物理结构上附加了插入事务 ID（insert_id）和删除事务 ID（delete_id）。
- **读路径无锁**：读取事务比较自身的 Read Epoch：如果某行被插入且 `insert_epoch <= read_epoch`，且没有被删除或 `delete_epoch > read_epoch`，则该行对当前事务可见。
- **性能优势**：写事务和读事务完全并发不冲突，读操作不需要申请任何物理读锁。

### Card 13: 逻辑删除元信息与版本链更新
- **逻辑删除**：在 DuckDB 中，删除行不是物理清除，而是将当前事务 ID 记录在行组的删除向量（Delete Vector）中，做逻辑标记。
- **更新操作**：更新被转化为“逻辑删除旧版本 + 插入新版本”，新旧版本通过物理版本链（Version Chain）级联。
- **退化隐患**：长事务不提交会导致大量的逻辑删除标记在内存中积压，拖慢后续扫描速率，需要定期回收。

### Card 14: 列式数据归并压实与数据压缩
- **物理优化**：当行组中的过期版本已超过全局 Safe Epoch 且无长事务访问时，DuckDB 会启动 Compaction。
- **压缩编码**：将未压缩的行数据利用 Run-Length (RLE)、Bit-Packing 或字典编码进行物理压缩，并写回只读 Column Block。
- **带宽损耗**：后台 Compaction 消耗 CPU 和磁盘 I/O 资源，在超大并发写入时可能短暂造成 QPS 的波动。

### Card 15: ACID 事务提交与 WAL 回滚截断
- **物理落盘**：事务提交时，内存中的脏元数据段和 WAL 物理块必须安全刷盘以完成持久化。
- **回滚操作**：如果事务 Abort，直接清理该事务的临时内存缓存并废弃 WAL 段，逻辑删除向量中的事务 ID 做失效标记。
- **设计折衷**：为了绝对 ACID 安全，多线程提交必须通过 Transaction Manager 执行排他锁，对极限短事务写入吞吐有一定制约。

## M4: 并行计算与查询管线

### Card 16: Push 模式执行引擎管线
- **数据流向**：传统的 Pull 模型由父算子调用子算子的 `next()`。DuckDB 采用 Push 模型：由底层的 Scan 算子将数据 Vector 推送到流水线（Pipeline）中，后续算子直接处理并下推。
- **并发优势**：Push 架构天然易于实现多物理流水线的水平分支执行，且能更加精准地将数据流推送至对应的物理 CPU 核心线程。
- **复杂性**：控制流与数据流相反，增加了调度框架和慢算子背压限流实现的复杂度。

### Card 17: 并行扫描与线程任务窃取
- **工作池调度**：利用 Task Scheduler 维护一个全局工作线程池（通常对应物理 CPU 线程数）。
- **任务划分**：表的多个行组（Row Groups）被划分为独立的扫描 Task。各线程执行物理 Scan。如果某线程空闲，会去其他繁忙线程的就绪队列中窃取（Work Stealing）扫描任务。
- **折中优化**：消除了线程间的饥饿问题，但在数据分布极度倾斜下，线程间频繁的任务锁竞争会带来轻微的开销。

### Card 18: 内存型哈希连接算法
- **向量化 Hash Join**：实现两表关联时，先读取 Build 侧的数据在内存中构建哈希表，然后利用 Probe 向量批量比对。
- **缓存敏感**：哈希表数据结构经过精细的缓存对齐，并使用 Selection Vector 来打包输出匹配的主键。
- **物理局限**：如果 Build 侧数据超出物理内存 hard limit，必须分治切片并执行落盘排序。

### Card 19: 外部归并排序与内存超限溢出
- **外部排序**：当对超大表执行 `ORDER BY` 且数据容量超出 Memory Limit 时，DuckDB 启用外部归并排序。
- **落盘排序块**：在内存中对各列数据 Vector 排序生成临时的排好序的 Blocks 并写入溢出区，最后在磁盘上进行多路归并排序合并输出。
- **性能滑坡**：排序速度会受限于物理磁盘的写入与读取 I/O 带宽，速度相比纯内存排序下降一到两个数量级。

### Card 20: 内存超限磁盘溢出机制
- **溢出控制器**：全局 Memory Manager 监控当前进程的总内存分配。
- **数据降级**：当内存触顶时，强行终止当前活跃 Hash Join 或 Aggregator 的内存扩展，将数据序列化压缩后写入临时的 `.tmp` 物理文件中。
- **运行保障**：避免了在处理超大规模复杂分析查询时直接遭遇 OS OOM 被强退，但付出了巨大的磁盘 I/O 代价。

## M5: 数据格式与物理接口

### Card 21: Parquet 数据源零拷贝解析
- **直接映射**：由于 Parquet 文件本身就是高度优化的列存结构，DuckDB 内部的 Parquet Reader 可以直接将其物理 Block 拷贝或映射 to 内存的 Vector 中。
- **过滤下推**：将 SQL 过滤条件直接下推至 Parquet 的 Row Group 级元数据，避免不合规 Parquet 数据的解压和内存拷贝。
- **适用场景**：在大厂混合数仓交互分析中，可直接在 S3 的 Parquet 数据上跑 SQL 关联，无需经过 ETL 导入。

### Card 22: Arrow 内存格式无缝映射
- **零拷贝桥接**：Apache Arrow 规定了业界列式内存的通用规范。DuckDB 对 Arrow 提供原生的 API 接口支持。
- **运行机理**：可以直接把 Arrow Record Batch 的内存物理地址作为 DuckDB 的数据源。由于两者的内存列式排布几乎一致，读取完全不涉及拷贝。
- **设计折衷**：支持这一特性需要额外的兼容性内存解析逻辑，但换取了与 Python 数据科学生态系统的毫秒级数据流通。

### Card 23: CSV 与 JSON 极速并行解析器
- **并行扫描器**：DuckDB 编写了专用的并行 CSV Scanner。它可以在多个 CPU 核心上并行定位 CSV 的换行符并分割扫描。
- **自适应类型推断**：前几批 Vector 会自动探测列的物理类型，并在读写中将推断类型应用于后面的行。
- **容错限制**：对于格式严重残缺或多边异构的非标准 CSV 文件，可能引发解析失败或频繁的回滚重试。

### Card 24: 嵌入式库架构与线程安全保证
- **单进程隔离**：DuckDB 核心设计为 C++ 进程内链接库，通过 API 嵌入 Python/Node/Go 中。
- **线程安全设计**：Catalog 目录管理、事务管理和内存池均包含细粒度的互斥锁（Mutex）保护，支持在多线程应用程序中并发查询同一数据库文件。
- **单点局限**：一个数据库文件在同一时刻只允许一个进程持有写连接，不能像传统的网络数据库那样进行多端并发物理写入。

## M6: 系统扩展与性能遥测

### Card 25: 元数据目录树管理
- **Catalog 架构**：Catalog 是维护数据库 Schema、表、视图、索引、类型和函数指针的底层数据结构。
- **版本化维护**：元数据变更（如 `CREATE TABLE`）同样受 Epoch MVCC 事务版本管理，允许不同事务在不同的 Schema 视图下并发隔离运行。
- **物理折衷**：高度的隔离保障了复杂系统修改的并发安全，但使得元数据访问路径长，增加了锁管理开销。

### Card 26: UDF 与插件扩展 SDK
- **扩展开发**：DuckDB 提供了高度解耦的 C++ 和 C 插件开发 API，允许开发者编写自定义函数（UDF）、新存储引擎或网络连接器。
- **向量化 UDF**：自定义函数支持输入 Vector 并输出 Vector，保证了用户扩展逻辑也能完美享受 SIMD 指令集加速。
- **设计折衷**：自定义插件在单进程内部直接访存，若插件代码写错发生 Segment Fault，会导致整个宿主主进程直接崩溃退役。

### Card 27: 执行计划 Profiling 遥测诊断
- **耗时追踪**：通过 `PRAGMA enable_profiling='json'` 可开启算子级性能剖析。
- **监控指标**：记录每个物理算子（如 HashJoin, PhysicalScan）的执行时长、处理的向量总数、临时磁盘溢出量以及线程活动热度。
- **诊断用途**：在执行大分析 SQL 性能变慢时，导出分析图谱，精准找出倾斜算子或没有走 Zone Map 跳过的瓶颈点。

### Card 28: 资源配额受限与内存硬上限
- **配额管理**：可以通过设置 `SET max_memory='32GB'` 强制限制当前 DuckDB 连接的物理运存开销。
- **主动干预**：一旦查询需要的临时缓冲区触及限制，系统会立刻拦截并优先触发数据落盘；对于无法落盘的查询（如某些不支持外排的复杂 UDF），自动抛出异常退出。
- **折中优化**：在多租户环境或内嵌于移动端设备时，能绝对规避因单条慢 SQL 恶意榨干宿主内存引发的大面积雪崩。
