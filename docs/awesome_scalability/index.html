M1.1 扩展原则与模式

垂直扩展 (Scale-up) 与水平扩展 (Scale-out) 的物理边界与演进规律

  • 技术机制: 垂直扩展（Scale-up）通过提升单机硬件（如 128 核 CPU、2TB 内存）提升吞吐，受制于摩尔定律及主板总线物理极限，且成本呈非线性飙升。水平扩展（Scale-out）通过廉价 PC 节点构建无共享集群，利用一致性哈希或哈希槽路由将负载均摊。
  • 应用策略: 在演进初期采用 Scale-up 提速，在核心写入吞吐或高可用度要求临界点时，全面向 Scale-out 无共享（Share-Nothing）架构迁移。
    • M1.2 扩展原则与模式

    无状态服务 (Stateless Services) 的共享无状态设计与会话共享机制

  • 技术机制: 要求应用层节点不保存任何客户端上下文（如 HTTP Session）。用户的会话状态外置存储到 Redis 集群或数据库中，客户端请求携带 JWT 或 Token，负载均衡器（如 Nginx/LVS）可以随机或以轮询算法分发请求到任意实例，实现节点的任意启停。
  • 应用策略: 将应用层业务逻辑与数据存储层彻底分离，是微服务实现秒级自适应弹性伸缩（HPA）的根本物理前提。
    • M1.3 扩展原则与模式

    共享无脑 (Shared Nothing) 架构在分布式计算与存储中的分流隔离性

  • 技术机制: Shared Nothing 架构下，每个节点拥有独立的 CPU、内存与磁盘，节点间仅通过高速网络交换数据，不存在物理共享资源，避免了分布式锁的全局争抢和互斥瓶颈。
  • 应用策略: 分布式存储系统（如 Cassandra、ClickHouse Cluster）采用此架构以获取近乎线性的扩展吞吐曲线。
    • M1.4 扩展原则与模式

    边缘计算 (Edge Computing) 与 CDN 动态/静态资源按区域就近分流

  • 技术机制: 静态资源（HTML、图片）通过边缘节点缓存，利用 Anycast IP 广播与 GeoDNS 将请求路由至最近的 CDN Edge。对于动态数据，采用边缘计算（如 Cloudflare Workers）执行前置鉴权与缓存裁判定向，稀释压向数据中心的流量。
  • 应用策略: 在全球化高流量场景下，至少应有 80% 以上的只读请求在边缘层被拦截处理，保护核心数据中心。
    • M2.1 数据库分片与路由

    数据库主从复制读写分离下的同步延迟与“写后立即读主库”策略

  • 技术机制: 读写分离架构中，主库处理写并产生 Binlog，从库拉取 Binlog 异步回放。在高并发写时，同步延迟（Replication Lag）可达秒级。若此时读从库会获取到陈旧数据。
  • 应用策略: 对于敏感操作（如新建订单后跳转查看详情），路由组件必须将请求强制定向至主库；或在客户端写入后打上版本标记，比对从库回放时间戳（TSO）无滞后时才允许读取。
    • M2.2 数据库分片与路由

    水平分片 (Sharding) 的分片键 (Shard Key) 选择偏差与散列不均

  • 技术机制: Sharding 将单表数据按行切分到不同物理库。Shard Key 决定路由方向。若选择不当（如按租户 ID 分片，而某头部租户数据量占 90%），会导致严重的数据和热点倾斜。
  • 应用策略: 应选用基数大、散列均匀且高频作为查询条件的字段（如 user_id）作为 Shard Key，并配合虚拟槽（Virtual Buckets）打散极值。
    • M2.3 数据库分片与路由

    分布式跨分片关联 (JOIN) 消除：反范式设计与二次查询归并路由

  • 技术机制: 物理分片后，跨分片的 JOIN 操作无法执行。解决策略有二：一是反范式设计，在订单表中直接冗余商品名称与用户名，消除 JOIN；二是二次查询路由，中间件在写端分布式查询子表并取回主键，再在内存中执行 Hash Join 归并。
  • 应用策略: 核心 OLTP 系统中应强制禁止跨库 JOIN，通过业务层数据组装或 Elasticsearch 同步实现复杂多维查询。
    • M2.4 数据库分片与路由

    分布式全局主键：Snowflake 雪花算法时钟回拨异常与自增序列性能

  • 技术机制: Snowflake 产生 64 位有序 ID（时间戳 + 工作节点 ID + 自增序列）。由于严重依赖系统时钟，一旦 NTP 服务发生时钟回拨，将导致 ID 碰撞。
  • 应用策略: 在算法中引入备份历史时间戳，若回拨差值小则自旋等待，差值大则切换备用 Machine ID 或直接抛出异常，或引入基于 Raft 的集中式分配。
    • M2.5 数据库分片与路由

    多数据中心多活 (Active-Active) 架构下的双向复制与数据冲突判定

  • 技术机制: 异地多活下，多数据中心同时接收写入，双向复制由于光速延迟必然导致冲突。判定算法包括：最后写入获胜（LWW，依赖时钟同步）、向量时钟（Vector Clocks，记录因果因果依赖）、CRDT（冲突无化解复制数据类型，如自增计数器）。
  • 应用策略: 业务设计上应尽量避免异地并发修改同一实体，对转账等高风险操作实施“单元化路由（Cell-based Routing）”，确保单一用户请求被锁定在固定数据中心。
    • M3.1 事件驱动与 CQRS 架构

    CQRS (命令查询职责分离) 读写模型物理隔离与最终一致性窗口

  • 技术机制: 写入端（Command）采用强范式、原子事务确保业务边界完整性；读取端（Query）根据页面展示高度定制。两者采用不同数据库，写端通过 Event Store 发送领域事件，读端消费事件并异步更新其只读结构。
  • 应用策略: 适用于读写比例极度失衡（如 100:1）且读需求多变的系统，需接受读端与写端之间存在毫秒级的最终一致性窗口。
    • M3.2 事件驱动与 CQRS 架构

    事件溯源 (Event Sourcing) 的事件日志追加写与查询重建状态回放

  • 技术机制: 不存储实体的当前状态，仅将实体的历史修改动作以“只追加写（Append-Only）”的事件流日志持久化。当需要重建状态时，从初始状态起按顺序依次“重放（Replay）”事件。为优化性能，会定期产生实体状态快照（Snapshot）。
  • 应用策略: 提供天然的审计日志追踪和历史状态回退能力，是高可靠金融交易系统和 DDD 模式的理想数据方案。
    • M3.3 事件驱动与 CQRS 架构

    事件驱动架构 (EDA) 下的发布订阅模式与松耦合微服务拓扑

  • 技术机制: 微服务间不通过强阻塞的 HTTP/RPC 调用，而是通过向事件总线（Event Bus）广播事件。上游不知道也不关心谁在消费事件，下游微服务自适应订阅事件并做出相应处理。
  • 应用策略: 大幅降低微服务间的拓扑依赖度，防止单个服务宕机导致整条调用链挂掉，提升了整个系统的水平弹度。
    • M3.4 事件驱动与 CQRS 架构

    领域驱动设计 (DDD) 中的限界上下文与事件风暴限界划分

  • 技术机制: 将复杂软件系统拆分为自治的限界上下文（Bounded Context），上下文内部模型保持强一致。利用“事件风暴（Event Storming）”会议，识别出核心领域事件（Domain Events）与命令（Commands），作为微服务拆分的物理边界。
  • 应用策略: 避免了传统微服务因强行按表拆分导致的“分布式单体”泥潭，实现了高内聚低耦合。
    • M4.1 异步消息传递与队列

    消息队列在海量流量突发下的削峰填谷与流量整形缓冲作用

  • 技术机制: 突发高并发请求（如整点抢购）在网关层转化为消息，直接写入具备高持久化吞吐的消息分区（如 Kafka/RocketMQ）。下游消费集群按照自身物理水位，以稳定的速率拉取（Pull）并消费，防止尖锐洪峰压垮数据库。
  • 应用策略: 将瞬时大流量积压转化为可控的队列延迟，在系统可用性与实时性响应间做出架构折衷。
    • M4.2 异步消息传递与队列

    最多一次、最少一次与精确一次的交付语义保障与事务消息机制

  • 技术机制: At-most-once（不重试，有丢失风险）；At-least-once（带重试与确认，无丢失但有重复）；Exactly-once（最少一次 + 幂等重试或两阶段分布式提交事务消息）。
  • 应用策略: 生产环境下通常使用性能更佳的 At-least-once 交付，然后强依赖消费端的幂等控制来模拟 Exactly-once。
    • M4.3 异步消息传递与队列

    消息幂等处理：唯一消息 ID、Redis 去重与数据库唯一约束防重入

  • 技术机制: 发信端对每条消息附加全局唯一 message_id。消费端在处理前，将该 ID 作为 key 存入 Redis 并设置过期时间（利用 SETNX 获取锁），或者作为主键写入“去重表”。若操作失败，说明消息已被消费，直接向队列返回 ACK 并丢弃。
  • 应用策略: 属于支付结算、库存扣减等异步事件消费场景的核心安全防线，杜绝网络抖动重试导致多扣款。
    • M4.4 异步消息传递与队列

    队列积压下的主动回压 (Backpressure) 与生产者发送速率动态反馈

  • 技术机制: 当下游消费者变慢导致中间件积压量暴涨时，消息队列自适应减小 TCP 接收滑动窗口，或者通过协议层发送暂停信令，迫使生产者减速或抛出过载异常。
  • 应用策略: 防止队列积压耗尽中间件的内存或磁盘空间，从而避免消息服务器级联崩溃。
    • M4.5 异步消息传递与队列

    死信队列 (DLQ) 的消息重新投递阈值、异常隔离与告警人工处置

  • 技术机制: 若某条消息发生业务格式异常，或者消费失败重试达到设定的阈值（如 16 次），队列中间件会自动剥离该消息并发送至死信队列（Dead Letter Queue），防止坏消息卡死整个分区的消费流水线。
  • 应用策略: 必须对死信队列设置高优先级指标监控，通过分析 DLQ 增长率可精准感知上游系统改版引入的代码缺陷。
    • M5.1 系统容灾与网格舱壁

    熔断器模式的三态状态机转换、探测窗口与故障自愈冷却机制

  • 技术机制: 熔断器监控调用失败率。
  • Close：正常放行。
  • Open：当滑动窗口内失败率超过阈值，转换为 Open，直接阻断调用并秒级返回 Fallback。
  • Half-Open：经过冷却期后转换，放行探测流量，若全部成功则合回 Close，若失败则退回 Open。
  • 应用策略: 核心容灾屏障。对于不影响核心支付流程的边缘调用（如用户推荐），必须设置熔断降级。
    • M5.2 系统容灾与网格舱壁

    令牌桶与漏桶限流算法差异：突发流量平滑度与排队排阻决策

  • 技术机制: 漏桶（Leaky Bucket）以恒定速率滴出请求，能产生绝对平滑的流量，不支持突发；令牌桶（Token Bucket）以恒定速率注入令牌，支持短时并发瞬间消耗桶内积存的令牌，允许突发（Burst）。
  • 应用策略: API 网关入口处通常选用令牌桶以应对合理的网络波动，而底层有严格 QPS 限制的第三方接口则倾向使用漏桶。
    • M5.3 系统容灾与网格舱壁

    舱壁隔离 (Bulkhead) 模式的线程池隔离、信号量限额与故障级联控制

  • 技术机制: 借鉴泰坦尼克号水密舱壁设计。如果所有微服务调用共享单一全局线程池，某个慢服务的请求积压会占满所有线程，导致其他服务无法处理。舱壁隔离为每个微服务分配独立的专有线程池或设置并发信号量上限。
  • 应用策略: Hystrix/Resilience4j 的标配。确保即便某一外部接口慢死，也不会波及其他服务。
    • M5.4 系统容灾与网格舱壁

    微服务网格 (Service Mesh) 的 Sidecar 模式、控制面与数据面解耦

  • 技术机制: 将微服务间的路由、熔断、遥测、安全通信（mTLS）等通用治理逻辑从代码中剥离，沉淀至独立的 Sidecar 进程（如 Envoy）中。Sidecar 作为数据面，控制面（如 Istio）动态下发配置参数。
  • 应用策略: 实现了服务治理逻辑与业务代码的完全解耦，使多语言微服务群落的统一管控成为可能。
    • M5.5 系统容灾与网格舱壁

    超时与指数退避重试 (Exponential Backoff with Jitter) 防重试风暴

  • 技术机制: 重试时，重试间隔应呈指数级拉大（如 $1\text{s}, 2\text{s}, 4\text{s}, 8\text{s}$），并且必须加入随机扰动时间（Jitter），打散大量客户端同时重试的周期性洪峰。
  • 应用策略: 在大型分布式系统的所有 RPC 与客户端连接层必须配置 Jitter，防范故障恢复后引发重试风暴（Thundering Herd）。
    • M6.1 大厂架构实践案例

    Netflix 混沌工程 (Chaos Engineering) 的 Chaos Monkey 随机注入与系统免疫

  • 技术机制: 核心思想是“在生产环境中主动且安全地制造混乱”。通过 Chaos Monkey 等工具，在工作日随机关闭线上服务器、断开网络链路或注入高延迟，检验系统容错自愈能力。
  • 应用策略: 混沌工程能够主动暴露未经过大促检验的架构死角，推动开发人员落实高可用防护。
    • M6.2 大厂架构实践案例

    Twitter 极速推文时间线 (Timeline) 拉取 (Pull) 与推入 (Push) 的写扩散折衷

  • 技术机制: 典型的高并发读写平衡设计。
  • 拉模式 (Pull/读扩散)：发文时仅写作者发件箱；粉丝读推线时去关注列表的所有发件箱合并。
  • 推模式 (Push/写扩散)：发文时将内容异步投递到所有粉丝的收件箱。对于名人（粉丝上亿），写扩散会引发写风暴，需采用混合模式（名人发推用拉模式，普通人发推用推模式）。
  • 应用策略: 针对海量社交网络和时间线场景，必须基于用户社交图谱的基数分布设计混合路由。
    • M6.3 大厂架构实践案例

    Amazon DynamoDB 无主多活复制、向量时钟 (Vector Clocks) 与 Read/Write Quorum

  • 技术机制: DynamoDB 是分布式 AP 系统的开山之作。采用 Masterless 架构，无单点主库。写入时使用 Vector Clocks 记录修改路径。通过 Read/Write Quorum 判定数据一致性：设定 $N$ 为副本数，$R$ 为读节点数，$W$ 为写节点数。当满足 $R + W > N$ 时，能保证每次读操作都能获取到最新的写入数据。
  • 应用策略: 适用于需要跨全球多活写入且可用性要求极高的场景（如购物车系统），通过 Quorum 灵活调整一致性强度。
    • M6.4 大厂架构实践案例

    Google Spanner 原子钟 + GPS (TrueTime API) 下的全球分布式事务强一致性

  • 技术机制: Spanner 实现了全球分布下的分布式强一致性事务。传统一致性算法受制于光速导致的物理时钟漂移。Spanner 在数据中心内部署了专有的 GPS 接收器和原子钟，提供 TrueTime API，可获取一个极窄的时钟偏差范围 $[t.\text{earliest}, t.\text{latest}]$（通常小于 7 毫秒），并在此时间差窗口内通过等待时钟前进而实现基于“真实时间戳（Commit Wait）”的多版本并发控制（MVCC）。
  • 应用策略: 全球强一致性多数据中心分布式数据库的终极形态，消除了分布式两阶段提交中的不确定性读问题。
    • M6.5 大厂架构实践案例

    Uber 基于 Sagas 模式的分布式多段柔性事务补偿机制与最终一致

  • 技术机制: 在微服务长事务场景下，传统的 2PC 容易因长时间锁死资源导致高可用度暴跌。Saga 将长事务拆分为一系列本地事务 $T_1, T_2, \dots, T_n$。若执行到 $T_k$ 失败，则 Saga 协调器（Orchestrator）反向顺序触发对应的补偿事务 $C_{k-1}, \dots, C_1$ 进行回滚，每个补偿操作必须支持幂等重试。
  • 应用策略: 广泛用于跨越多个自治服务的复杂业务场景（如打车出行中的约车-扣款-派单流程），牺牲瞬时一致性以换取高吞吐率。
