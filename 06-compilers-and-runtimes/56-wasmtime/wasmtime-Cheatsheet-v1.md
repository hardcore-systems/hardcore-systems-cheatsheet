# Wasmtime 高效 JIT 虚拟机与轻量沙箱运行时速查海报

## J-Ladder 阶梯逻辑模型

### L0 一句话本质
Wasmtime 的本质是通过 JIT 编译器（Cranelift）将 WebAssembly 虚拟字节码即时编译为本地平台的高效物理机器码，并通过操作系统虚拟地址空间及 Guard Pages（保护页）硬件机制，在用户态构建零额外检查开销的强隔离轻量级安全沙箱运行时。

### L1 四句话逻辑
1. **静态验证与 SSA 转换**：在加载时对 Wasm 字节码进行严苛的类型与控制流静态验证，转换为 Cranelift 特有的 SSA（单静态赋值）中间表示。
2. **零开销硬件边界防护**：通过预留 4GB+2GB 的巨大虚拟内存虚拟地址空间（不占用物理内存）配合边界 Guard Pages，利用物理 CPU 页保护中断替代运行时的越界指令检查。
3. **基于组件的高阶语义互通**：利用 WIT 接口定义语言与组件模型（Component Model），实现复杂对象和异步语义（WASI 0.3）在 Host 与 Guest 之间的强类型安全转换。
4. **精确的中断与陷阱逃逸**：依托信号处理器（Signal Handler）捕获物理硬件异常（如除零、页错误），结合栈回溯（Stack Unwinding）将其安全映射回运行时的 Wasm Trap 机制，防止宿主进程崩溃。

### L2 核心数据流转拓扑
`Wasm 字节码加载` ➜ `Cranelift JIT 编译 (SSA CLIF ➜ 物理机器码)` ➜ `内存虚拟分配 (4GB + Guard Pages)` ➜ `Host 调用 Linker 注入导入项` ➜ `Instance 实例化并运行` ➜ `Wasm 执行越界访问` ➜ `物理 CPU 触发 Page Fault 异常` ➜ `Signal Handler 捕获并解析为 Wasm Trap` ➜ `安全回溯到宿主环境`

## 📂 核心知识卡片 (Cards 1-28)

### Card 1: Wasmtime Engine 与 Config 配置
*   **核心原理**: 编译和运行策略的配置底座。Engine 是编译和缓存配置的全局共享容器，而 Config 则是初始化配置的非同步构建载体。
*   **技术细节**: Wasmtime 通过 `Config` 指定是否启用 JIT (Cranelift)、AOT、编译优化级别（None/Speed/Size）以及虚拟内存防护参数。一旦 Config 被传递给 `Engine::new(config)`，整个运行时策略便在内存中不可变冻结，并作为缓存和实例生成的全局上下文在多个 Thread 间共享。
*   **折中与防范**: 引擎层面的激进优化（如 Speed 优化等级 2）会大幅拉长首次加载和即时编译的开销。对于高频启动的冷服务，应开启 AOT (Ahead-of-Time) 编译并使用本地缓存以节省冷启动延时。

### Card 2: Store 状态空间与数据隔离
*   **核心原理**: 包含运行期数据隔离的物理沙箱边界。每个 Store 封装了一组独立的 Wasm 实例状态、宿主自定义 State、垃圾回收根集以及资源限额（Limits）。
*   **技术细节**: Wasmtime 遵循“无共享内存（No Shared Memory）”隔离设计，不同 Store 之间的实例无法直接访问对方的线性内存。宿主与 Store 之间通过 `StoreContext` 安全交互。Wasm GC 对对象进行跟踪时，Store 作为垃圾回收扫描的根集来源。
*   **折中与防范**: Store 是单线程绑定的，非 `Send`，这限制了单个 Store 跨物理核心并发调度。对于多核并发应用，宿主必须为每个工作线程创建独立的 Store，并在应用层做状态协调。

### Card 3: Module 静态验证与 JIT 编译
*   **核心原理**: 保证执行安全性与本地优化的安全编译单元。从 Wasm 字节码到 ELF 格式二进制机器码的并行流水线。
*   **技术细节**: `Module::new` 首先进行单遍词法静态验证，确保控制流跳转安全、堆栈类型匹配，不包含危险指令。验证通过后，将 Wasm 模块切分为多个编译单元，由多个后台线程并发调用 Cranelift JIT 编译器进行 SSA IR 优化和机器码翻译，最终在内存中拼装成包含重定位表的 ELF 结构。
*   **折中与防范**: 静态验证和 JIT 编译属于 CPU 密集型任务，会对宿主服务的吞吐造成剧烈波动。可通过开启异步预编译机制，预先将 Module 编译成 `AOT` 二进制并在磁盘上持久化缓存。

### Card 4: Linker 链接器与导入路由
*   **核心原理**: 虚拟机安全导出与宿主接口注入的“纽带”。在实例化前解析并填补 Module 声明的所有导入项（Imports）。
*   **技术细节**: `Linker` 维护一个从模块名和函数名映射到内部导出对象的类型安全哈希路由表。当实例化模块时，Linker 自动寻找并校验对应的 Wasm 函数、宿主回调（Host Functions）以及共享内存对象，建立调用路由表。
*   **折中与防范**: 每次实例化时的动态链接会引入一定的哈希查找开销。可以通过在 Linker 中使用静态预链接（Pre-linking）生成专用的实例化模板来抹平这部分运行时损耗。

### Card 5: Instance 实例化生命周期
*   **核心原理**: 独立执行引擎的瞬时激活。为 Module 绑定具体的运行时 Store 上下文，并初始化内存与启动入口。
*   **技术细节**: 实例化过程中，Wasmtime 为 Instance 分配虚拟内存页（Memory）、初始化全局变量、导入宿主函数，并将 Module 内定义的 Data 段拷贝进线性内存。最后，若模块中包含 `start` 入口函数，会立即调用并执行它，完成冷启动初始化。
*   **折中与防范**: 实例化包含系统调用（如内存映射 `mmap`）以及大量内存拷贝，开销相对偏重。推荐使用 Wasmtime 的虚拟内存池机制预分配物理页，降低每次实例化带来的上下文切换延时。

### Card 6: Host-to-Guest Trampoline 蹦床调用
*   **核心原理**: 抹平宿主 ABI（C 语言/Rust 规范）与 Wasm JIT 特殊寄存器传参规范差异的调用适配层。
*   **技术细节**: Wasmtime 虚拟机内部生成的机器码采用高效但特殊的寄存器和栈帧传参规则（Guest ABI）。当宿主通过 Rust API 触发 `func.call()` 时，控制流会先进入一段由汇编编写的 `Trampoline`（蹦床）代码。蹦床函数负责将标准的宿主参数数组（或 Rust 类型）打包，压入对应的 CPU 寄存器，调整栈对齐，最后执行 `jmp` 跃迁进 Wasm JIT 编译出的物理地址。
*   **折中与防范**: 蹦床调用会引入微秒级的跳转跳转损耗和寄存器状态压栈开销。为了性能，高频调用的宿主接口推荐使用 `TypedFunc` 强类型包装，允许编译器在编译期内联蹦床适配。

### Card 7: Cranelift SSA CLIF 中间表示
*   **核心原理**: 基于静态单赋值（SSA）形式的平台无关编译器中端表示。
*   **技术细节**: Cranelift 的 IR（CLIF）以块（Blocks）为基本控制流单位，每个块包含一系列 SSA 指令。变量只能在定义时被赋值一次，多路径汇合时的值流转采用显式的块参数（Block Parameters）进行传递，彻底避免了传统虚拟寄存器的冲突分析。
*   **折中与防范**: SSA 形式在分析和优化（如死代码消除）上非常高效，但它相比非 SSA IR（如栈式 IR）需要占用更多的内存来维护值定义链。Cranelift 内部通过高度紧凑的 Arena 向量存储索引以压榨内存损耗。

### Card 8: Regalloc2 寄存器分配器
*   **核心原理**: 为 SSA 中海量的虚拟变量分配合理物理寄存器的极致引擎。
*   **技术细节**: Wasmtime 采用最新的 `Regalloc2` 库。它使用基于 SSA 树结构的一步式扫描算法，系统性地权衡虚拟变量的生命周期与冷热路径。在寄存器不足时，插入最少量的栈溢出（Spill）与寄存器重加载（Reload）机器指令。
*   **折中与防范**: 寄存器分配是 JIT 编译中最耗时的一步。Regalloc2 针对此进行优化，在分配质量与编译速度之间进行了深度平衡，相比 LLVM 的寄存器分配，它速度快一个数量级，但极端复杂代码的分配紧凑度略逊。

### Card 9: Wasm 字节码即时翻译循环
*   **核心原理**: 快速、单遍扫描的 Wasm 虚拟指令翻译器。
*   **技术细节**: 在将 Module 编译时，Cranelift 的底层翻译器会顺序扫描 Wasm 字节码流。由于 Wasm 采用栈式虚拟机（Stack-based VM）设计，翻译器在内存中维护一个轻量级的“操作数栈仿真表”。当遇到 `i32.add` 时，翻译器弹出仿真表顶部的两个虚拟值，并生成对应的 CLIF SSA 指令（如 `v2 = iadd v0, v1`）追加到当前 Block 尾部。
*   **折中与防范**: 单遍扫描翻译规避了构建庞大语法树的内存开销，但因为缺乏全局上下文视野，它生成的初始 CLIF 包含大量冗余，必须在后续阶段开启多轮轻量优化来精简机器码。

### Card 10: Cranelift 编译优化策略
*   **核心原理**: 快速且高效的本地代码精简策略。
*   **技术细节**: 针对 JIT 场景，Cranelift 摒弃了复杂的循环展开等重型编译器优化，专注于高收益的轻量级优化组合：常量折叠（Constant Folding）、公用子表达式消除（GVN）、死代码消除（DCE）以及局部的全局变量内联（Inlining）。
*   **折中与防范**: 激进的编译优化会严重损害 JIT 的即时编译性能，甚至抵消 Wasm 运行时的提升。Wasmtime 提供了编译级别选项，在非生产测试下建议配置为 `OptLevel::None` 以换取极致的冷启动加载速度。

### Card 11: 蹦床函数动态机器码生成
*   **核心原理**: 运行期为特定 Host 闭包或动态导出函数生成物理装载机器码的能力。
*   **技术细节**: 当宿主在 Linker 中动态注册一个闭包，且该闭包的参数签名在编译期无法确定时，Wasmtime 会在运行期借助 Cranelift 动态生成一段机器码蹦床。该机器码负责读取 Wasm 传递的物理寄存器参数，将其序列化为内存中的 Rust `Val` 数组，然后再回调给宿主的闭包。
*   **折中与防范**: 动态机器码生成涉及修改内存页权限（`mprotect` 读写 -> 可执行），频繁操作会引发显著的操作系统页表锁争抢。因此，动态闭包应在启动时集中链接，避免运行期动态创建。

### Card 12: JIT 编译产物反序列化缓存
*   **核心原理**: 零编译延时的重用机制。将 JIT 生成的本地机器码序列化为标准的 ELF 二进制文件并存储在磁盘上。
*   **技术细节**: Wasmtime 允许将 `Module` 序列化为磁盘文件。反序列化加载时，Wasmtime 利用 `mmap` 直接以“只读且可执行”的权限将该 ELF 文件装载进内存物理地址空间，由于不需要进行任何 Cranelift 编译步骤，加载耗时直接缩短至微秒级。
*   **折中与防范**: 序列化编译产物高度绑定当前的 CPU 架构与 Wasmtime 引擎版本。如果宿主升级了 CPU（如启用了新的 AVX 指令集）或 Wasmtime 版本，缓存必须作废并自动重建，否则会导致指令非法（SIGILL）崩溃。

### Card 13: Wasm 线性内存虚拟地址映射
*   **核心原理**: 实现 Wasm 沙箱内存安全的基础。将 Wasm 内部的地址空间（Linear Memory）与物理内存解耦。
*   **技术细节**: 当 Wasm 申请一片线性内存时，Wasmtime 会在虚拟内存中预留一大片连续的虚拟地址空间。在 64 位系统上，Wasmtime 默认使用静态内存策略：直接通过 `mmap`（或 Windows `VirtualAlloc`）申请 4GB 的匿名虚拟地址空间，但仅将实际的 Wasm 堆大小（如几十 MB）提交为物理页。
*   **折中与防范**: 预留 4GB 的虚拟空间不会占用真实的物理 RAM 内存。但这会严重消耗虚拟地址空间（Virtual Address Space）。对于拥有数十万并发实例的超多租户场景，这可能导致虚拟内存地址耗尽，需要将内存策略微调为“动态分配模式”。

### Card 14: Guard Pages 虚拟内存硬件防线
*   **核心原理**: 零软件开销的边界溢出检测。在 4GB 虚拟地址空间末端额外追加一片 2GB 的“不可读写且不可执行”的保护区（Guard Pages）。
*   **技术细节**: WebAssembly 指令访问内存时使用 `i32.load` 偏移寻址，最大偏移值为 4GB 加上指令静态偏移（通常在 2GB 内）。Wasmtime 通过在 4GB 虚拟内存的末端硬追加 2GB 的 Guard Pages，使得任何物理上可能发生的越界指针必然落在该保护区内。由于保护区在内核页表中被配置为无读写权限，越界访问会被物理 CPU 直接以 Page Fault 中断拦截。
*   **折中与防范**: 这种硬件防线彻底免除了编译器在生成的机器码中插入 `if addr >= bound` 的软件边界检查，使得寻址性能提升约 30%。缺点是单实例消耗的虚拟地址空间直接攀升至 6GB（4GB + 2GB）。

### Card 15: Page Fault 信号拦截与 Wasm Trap 转换
*   **核心原理**: 发生硬件越界时，安全逃离并防止宿主崩溃的自我治愈。
*   **技术细节**: 当 Wasm 代码越界访问 Guard Pages 时，物理 CPU 触发页保护异常，操作系统内核向当前进程发送 `SIGSEGV`（Linux）或触发 `SEH`（Windows）。Wasmtime 在启动时预先注册了全局信号处理器。当信号触发时，处理器捕获该信号，检查引发异常的指令指针（IP）是否落在 Wasmtime 生成的机器码区间内。若是，则安全修改当前线程的上下文堆栈，将其强行跳转回宿主的 Trap 处理逻辑，将底层的进程级 Segfault 软化并抛出为 Rust 的 `Wasm Trap: Out of bounds memory access` 错误。
*   **折中与防范**: 信号捕获机制高度依赖平台相关的底层硬件堆栈和内核中断，处理信号会引入数微秒的系统延迟。因此，Trap 机制仅适用于处理异常，不能用于正常的控制流转。

### Card 16: memory.grow 内存动态扩容
*   **核心原理**: 沙箱内部堆空间的自适应弹性收缩机制。
*   **技术细节**: 当 Wasm 代码调用 `memory.grow(pages)` 申请更多内存时，控制流通过 Trampoline 进入运行时。如果采用的是静态内存策略，Wasmtime 只需调用 `mprotect` 将 4GB 地址空间中对应新增部分的内存页权限由“无权限”切换为“读写权限”（物理内存页随后由操作系统按需动态提交），并更新 Wasm 实例内部的堆上限值。
*   **折中与防范**: 在动态内存策略下，如果物理扩容空间不足以保持内存连续，Wasmtime 被迫在物理堆中重新分配一片更大的内存区并执行大内存块拷贝（Memory Copy），这会导致短暂的运行期卡顿。

### Card 17: 沙箱 OOM 硬件级控制
*   **核心原理**: 防止多租户模式下单个恶性实例通过大扩容把宿主服务器物理内存耗尽的防御策略。
*   **技术细节**: 宿主可以在实例化时，通过 `ResourceLimiter` 接口为 Store 绑定最大内存上限限制。当 Wasm 执行 `memory.grow` 尝试突破该上限时，Wasmtime 会直接拦截并返回 `-1`（Wasm 规范中的内存增长失败标识），强制 Guest 代码抛出内部 OOM 异常或自行释放，而宿主本身的物理内存安全毫发无损。
*   **折中与防范**: 部分粗糙的 Wasm Guest 编译物（如没有处理分配失败的 C 代码）在 grow 返回 -1 时会立即陷入不可恢复的死循环或引发崩溃 Trap。宿主应结合 epoch 指令超时限制来防止此类实例发生挂起。

### Card 18: 虚拟内存池多租户复用
*   **核心原理**: 为海量瞬时启动的微服务实例而设计的预分配物理内存池。
*   **技术细节**: 在高并发多租户环境下，频繁地为实例调用 `mmap` 和 `munmap` 会产生显著的内核页表锁竞争（Page Table Lock Contention）。Wasmtime 提供了 `InstanceAllocationStrategy::pooling` 选项。运行时在启动时一次性从 OS 申请一大片连续的虚拟内存区域，划分为成百上千个固定大小（如 6GB）的 Slot。每次实例化时，Wasmtime 仅需从池中租借一个空闲 Slot，避免了任何运行期的系统调用。
*   **折中与防范**: Pooling 策略需要在启动时占用数 TB 的虚拟地址空间（即使没有物理分配），因此该策略仅能在 64 位系统上正常运行，且每个 Slot 的内存上限必须在启动时静态指定。

### Card 19: 多线性内存 (Multi-Memory) 支持
*   **核心原理**: 现代 WebAssembly 规范中的“内存多段”设计，允许单个实例内拥有多个独立的内存地址空间。
*   **技术细节**: 传统 Wasm 实例仅能操作一个名为 `memory` 的地址空间（Index 0）。Wasmtime 实现了多内存提案，允许实例同时读写多个命名内存（如 `memory.get 1`）。这对于跨模块数据传递非常高效，可以通过 Wasm 内置的 `memory.copy` 指令实现两个独立内存沙箱之间的高性能无拷贝或原子内存数据迁移。
*   **折中与防范**: 多个内存意味着单个 Instance 将消耗数倍的 Guard Pages 和 Slot 资源，这对内存池（Pooling）的规划和虚拟地址空间管理提出了更高的对齐挑战。

### Card 20: WASI Preview 1 系统调用封装
*   **核心原理**: 让 Wasm 代码能够跨平台以安全受限的方式调用物理操作系统 I/O 接口的规范。
*   **技术细节**: WASI Preview 1 采用经典的文件描述符（File Descriptor）模型。Wasmtime 将操作系统底层的 I/O 接口（如读、写、打开文件、Socket）抽象并封装为 Wasm 导入项。Guest 编译出的 Wasm 可以通过执行 `wasi_snapshot_preview1::fd_write` 等指令安全发起系统调用，且调用链均在宿主的控制范围内。
*   **折中与防范**: 传统的 FD 模型在复杂的高阶抽象（如异步 IO、强类型管道、生命周期跟踪）上能力受限，因此 Wasm 社区已逐步将重心迁移至基于组件模型的 WASI Preview 2。

### Card 21: WASI Preview 2/3 组件模型演进
*   **核心原理**: WebAssembly 的下一代模块化与微服务互通规范。组件模型颠覆了传统的 Wasm 纯数值栈，引入了高阶类型与异步抽象。
*   **技术细节**: WASI Preview 2 基于 WebAssembly Component Model。它将两个组件之间的物理内存彻底隔离，通过通用的 WIT（WebAssembly Interface Type）接口定义语言建立通信管道。Wasmtime 提供了组件解析引擎，支持组件之间的动态依赖校验、高阶对象传递（如 String, Record, Resource）以及双向 Canonical ABI 解码。
*   **折中与防范**: 组件模型的解耦度极高，但不同组件在跨越调用边界时，涉及大量的内存数据深拷贝（Serialization/Deserialization）与 Canonical 压栈开销。

### Card 22: WIT 接口语言与 Canonical ABI 桥接
*   **核心原理**: 实现宿主复杂数据结构与虚拟机内部线性内存（Linear Memory）之间的安全边界转换。
*   **技术细节**: Wasm 内部只认识整数和浮点数，根本无法理解什么是“String”或“JSON”。当宿主向 Wasm 传递字符串时，Wasmtime 的 Canonical ABI 自动遵循标准：先在 Wasm 的线性内存中申请一块合适大小的空间，将宿主的 UTF-8 字节数据拷贝进去，然后将该内存的“首地址指针”和“字节长度”这两个 i32 整数作为物理参数传递给 Wasm 函数。
*   **折中与防范**: Canonical ABI 自动处理了这部分脏活，但这意味着高频传递大对象（如图像、复杂 JSON）会造成频繁的内存申请、数据拷贝和释放，对运行效率产生微弱的开销，需要结合共享缓冲区（Shared Buffer）设计做高级调优。

### Card 23: WASI 异步资源通道与 WASI 0.3
*   **核心原理**: 彻底解决 WebAssembly 在 JIT 沙箱内部执行网络请求或等待 I/O 时会导致宿主物理线程挂起（Blocking）的并发障碍。
*   **技术细节**: WASI 0.3 规范引入了原生的异步通道与协程语义。当 Wasm 执行非阻塞的异步网络读写时，Wasmtime 底层会捕获该操作并生成一个挂起的 Promise 资源槽。Wasm 实例的主执行流安全挂起（返回 Yield 标记），将宿主的执行权归还给外部的异步 Executor（如 Rust Tokio 运行期），直到宿主的 Mio 反应器监听到物理 I/O 就绪后，再自动推进该 Wasm 实例状态机。
*   **折中与防范**: 异步 WASI 的执行流非常复杂，要求 Wasm 编译产物本身也必须启用 async-component 代码生成逻辑，这会大幅增加生成的 Wasm 模块的字节码体积。

### Card 24: 虚拟文件系统与网络沙箱阻断
*   **核心原理**: 极致的文件与网络安全访问防护，确保恶性的恶意 Wasm 代码无法入侵宿主的主机文件系统和内部局域网。
*   **技术细节**: Wasmtime 并不直接允许 Wasm 使用真实的系统路径。宿主在配置 WASI 时，可以显式创建一个 `Dir` 映射，将 Wasm 的内部根路径 `/` 虚拟映射到宿主的具体安全目录（如 `/tmp/wasm_sandbox`）。Wasmtime 内部的系统调用拦截器在运行时会对任何文件操作进行“绝对路径规范化校验（Relative Path Resolve）”，凡是带有 `..` 尝试向上逃逸的系统调用都会被直接硬件截断并抛出权限 Trap。对于网络，Wasm 无法直接发起 Socket 连接，必须由 Linker 明确赋予其特定的端口和 DNS 绑定白名单许可。
*   **折中与防范**: 这种强隔离意味着宿主必须为每个实例精心配置所有必要的文件系统权限与网络路由许可，这显著提升了宿主的开发与运维配置复杂度。

### Card 25: Signal 陷阱路由控制与致命隔离
*   **核心原理**: 将虚拟机运行期由于除零、栈溢出等引发的致命 Wasm 故障，与宿主本身的正常逻辑彻底隔离。
*   **技术细节**: 虚拟机在执行 Wasm 指令时，可能会因为恶意输入触发除以零错误（Integer Division by Zero）或非法内存访问（SIGILL）。Wasmtime 通过在每个线程局部（Thread-Local）设置 Trap 上下文标志来路由。信号处理器监听到物理异常时，若 Thread-Local 标志显示当前正在执行 Wasm 代码，则将此信号转换为 Trap 并跳转回最近的宿主 await 边界；若发现异常发生在宿主自身的代码中，则交还给宿主默认处理机制（如崩溃 Panic），防止两者混淆。
*   **折中与防范**: 精确路由高度依赖线程局部的原子读写。在频繁切换执行环境的高并发场景下，过多的 TLS 访问会引入细微的分支寻址开销。

### Card 26: Backtrace 栈回溯分析与 DWARF 解析
*   **核心原理**: 将运行期 JIT 生成的裸物理机器码堆栈地址（Stack Frames），还原回可读的 WebAssembly 源码行列号。
*   **技术细节**: 当虚拟机抛出 Trap 异常时，Wasmtime 会捕获当前 CPU 的栈帧指针。由于 JIT 机器码中不包含原始函数名，Wasmtime 会在 JIT 编译时自动解析并保留 DWARF 符号表信息。通过比对引发异常的物理指令指针（IP）地址与 DWARF 中登记的映射范围，反向解算出当前栈帧对应的 Wasm 函数索引以及源代码行列号，在异常控制台输出漂亮的 Backtrace。
*   **折中与防范**: 在生产环境下，保留完整的 DWARF 符号表会显著膨胀 Module 的内存占用（可能膨胀数倍）。对于不需要调试的生产环境，可以显式调用配置关闭 `generate_debug_info` 以榨干内存。

### Card 27: Fuel 与 Epoch 协作式强占限额
*   **核心原理**: 解决用户态沙箱执行“死循环（Infinite Loop）”或超长计算任务时，导致宿主物理线程被强行独占（DOS 攻击）的监控机制。
*   **技术细节**: Wasmtime 提供了两种防饥饿限额策略：1. **Fuel（燃料）机制**：在 Cranelift 编译 JIT 机器码时，在每个 Block 的入口代码中自动插入一行原子扣减指令。宿主为 Store 注入一定的 Fuel 计数。当计数归零时，Wasm 抛出 `Trap::OutOfFuel` 强退。2. **Epoch 机制**：宿主启动一个独立的守护线程，以固定时钟周期（如 10ms）递增全局的 Epoch 计数器。Wasm 代码周期性轮询此版本号，一旦发现版本号落后设定阈值，立即 Yield 并归还执行权。
*   **折中与防范**: Fuel 机制是指令级的精确拦截，但由于需要在生成的机器码中频繁插入原子扣减与跳转指令，会导致 10% 左右的 Wasm 纯计算性能损耗；而 Epoch 机制是基于时钟采样的，性能损耗微乎其微（小于 1%），但在时间截断上存在毫秒级的精度延迟。

### Card 28: call_indirect 间接调用签名校验
*   **核心原理**: WebAssembly 函数表动态间接跳转时的安全哨兵。防范黑客利用内存篡改实现控制流劫持（CFI）。
*   **技术细节**: 当 Wasm 执行 `call_indirect` 指令根据索引动态跳转到 Table 中的某个函数时，编译器会在生成的物理跳转指令前插入一段硬校验。Table 中除了存储函数指针外，还额外存储了每个函数的签名哈希值（Signature Hash）。编译器会先提取跳转目标函数的哈希值，与调用处声明的期望签名进行 CAS 般比对，一致才执行 `call`，否则直接触发 Trap 强退。
*   **折中与防范**: 间接签名校验在每个动态跳转前引入了两次额外的内存寻址与一次分支比对。对于包含大量虚函数调用或动态分派的语言（如 C++ 编译成的 Wasm），高频的间接调用会带来微弱的跳转延迟。
