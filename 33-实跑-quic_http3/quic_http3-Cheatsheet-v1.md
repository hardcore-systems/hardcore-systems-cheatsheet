# h2o / quic-http3 高密度核心卡片系统 (中文版)

## L0 ~ L2 知识阶梯

*   **L0 一句话本质**: H2O 是一个极致优化的 HTTP/1/2/3 高性能服务器，它通过自研无锁 `evloop` 事件循环、基于 O(1) 的 HTTP/2 优先依赖树、以及紧密耦合的 `quicly` 传输层与 `picotls` 密码栈，在极低内存拷贝与 CPU 周期开销下实现高吞吐的网络流处理。
*   **L1 四句话逻辑**:
    1.  **自研事件循环与定制定时器轮**：不依赖庞大的通用库，基于 `h2o_evloop_t` 封装轻量级多路复用，使用专门优化的 `h2o_timerwheel_t`，确保海量连接下超时的增删操作达到 O(1) 物理复杂度。
    2.  **紧密耦合的模块化 QUIC/TLS 栈**：采用原生 `quicly` 作为传输引擎，并结合 `picotls` 实现 TLS 1.3 的零拷贝加解密；依靠自研的 AES-GCM 融合引擎加速数据包生成，达成极高的 HTTP/3 吞吐。
    3.  **零 HOLB 的 HTTP/2 优先级调度树**：在发送端构建严格的依赖优先级树，所有连接并发流的数据均被切分为 16KB 的分片，通过加权赤字轮询进行复用发送，从根本上防止单个大文件阻塞其它敏感资源。
    4.  **零拷贝 I/O 与 GSO 批量卸载**：全程使用 `h2o_iovec_t` 传递分段指针，配合 UDP 的分段卸载 (GSO) 机制，把数十个 UDP 数据包合并为一次系统调用发送，极大消除了用户态与内核态上下文切换的开销。
*   **L2 核心数据流转拓扑**:
    *   `Downstream Packet (UDP / TCP)` ➜ `Network Socket (Worker Thread)` ➜ `h2o_evloop_t` ➜ `quicly_conn_t (Decrypted by picotls)` ➜ `quicly_stream_t` ➜ `h2o_req_t / Core Handler` ➜ `h2o_http2_scheduler / h2o_http3_conn_t` ➜ `16KB Chunking / QPACK` ➜ `h2o_socket_write` ➜ `UDP GSO / sendmmsg` ➜ `Physical Network`.

## 📂 M1: 自研轻量级事件循环 (h2o_evloop) (Cards 1-4)

#### Card 1. h2o_evloop_t 的拓扑结构与基于 Epoll/Kqueue 的多后端抽象绑定
*   **技术机制**: H2O 自研的 `h2o_evloop_t` 结构体并不依赖第三方网络库。它在内部实现了一个轻量级的多路复用抽象层，支持根据编译时宏定义绑定不同操作系统的底层 I/O 后端（Linux 的 `epoll`、BSD/macOS 的 `kqueue`、Windows 的 `select`），屏蔽底层差异并实现极致的效率。
*   **应用策略**: 在生产环境中，H2O 通过检测操作系统内核版本，在编译期自动选择 `epoll` 或 `kqueue`，最大化并发吞吐性能。

#### Card 2. 延迟状态变更链表 `_statechanged` 与合并更新机制
*   **技术机制**: 当套接字因读写就绪或等待状态发生变更时，H2O 不会立刻调用 `epoll_ctl` 进行内核状态修改。而是将对应的 `h2o_socket_t` 挂载到事件循环的 `_statechanged` 单向链表中。只有在事件循环进入新一轮 Tick 的开头阶段，才会批量遍历并调用一次系统更新函数，大幅消除了上下文切换。
*   **应用策略**: 对于高吞吐的代理场景，这种合并更新能够有效降低在高并发包处理时的 CPU 自用开销。

#### Card 3. h2o_timerwheel_t 级联定时器轮常数时间增删机理
*   **技术机制**: H2O 在定时器处理中使用了 `h2o_timerwheel_t`（时间轮）。针对 HTTP Keep-Alive 大部分连接的超时周期完全一致的特点，时间轮将定时器按过期槽位分布在级联的桶（Buckets）中。每次循环只需要递增时钟游标并检查对应桶，添加与删除均是 $O(1)$ 操作，避免了红黑树结构随节点数增长而变慢的缺陷。
*   **应用策略**: 配置 `keepalive-timeout` 等宏观参数时，时间轮能自动对超长时间的空闲连接进行极其轻量的高效轮询清理。

#### Card 4. 单进程多线程工作模式下的套接字 SO_REUSEPORT 内核分发
*   **技术机制**: H2O 采用单进程多工作线程（Worker Threads）架构。每个 Worker 启动并拥有一个独立的 `h2o_evloop_t` 实例。为了防止多个线程在 accept 阶段竞争同一套接字产生锁冲突，H2O 开启了 `SO_REUSEPORT`，使得网卡收包后，由内核直接通过 Hash 算法对称地将新连接派发至指定 Worker 线程，实现彻底无锁。
*   **应用策略**: 生产环境将工作线程数 `num-threads` 设为物理核心数，确保各核心资源独立且负载均衡。

## 📂 M2: QUIC 传输层与 quicly 库集成 (Cards 5-9)

#### Card 5. quicly_conn_t 与 quicly_stream_t 状态机双重控制
*   **技术机制**: H2O 的 HTTP/3 底层通过子模块 `quicly` 驱动。`quicly_conn_t` 结构体承载整个 QUIC 传输连接上下文，负责连接级流控、加密握手、丢包恢复和拥塞算法演进；每个连接内划分出多个 `quicly_stream_t` 结构，专门用于多路复用的数据帧传递和流级重传。
*   **应用策略**: 监控系统中应对 `quicly` 的连接级健康度与活动流数进行差异化统计，以便诊断传输异常。

#### Card 6. picotls (TLS 1.3) 密钥衍生与 0-RTT 极速会话票据处理
*   **技术机制**: `quicly` 原生依赖极其紧凑的 TLS 1.3 实现库 `picotls`。握手阶段，`picotls` 在内存中就近解析 TLS Client Hello，生成对称工作密钥，并将 0-RTT 的 Session Ticket 信息进行无损恢复。使得再次建连的客户端能够在发送首个握手包时就携带加密数据，消除了握手的往返时延（RTT）。
*   **应用策略**: 在客户端部署中合理启用 session resumption，让再次请求达到零握手开销（0-RTT），提升首屏渲染速度。

#### Card 7. QUIC 握手阶段首包嗅探（First Byte Routing）与 CID 路由分发
*   **技术机制**: H2O 的 UDP 监听套接字收到物理包后，由 `h2o_http3_handle_packet` 函数进行首字节嗅探。若发现是新建握手包，则随机生成一个唯一的 Connection ID (CID) 并分发给客户端；若是已有连接的数据包，直接解析出 CID，并在全局无锁哈希表中检索对应的 `quicly_conn_t`，避免解析包头多余数据。
*   **应用策略**: 该机制确保了高频包率下连接分流路径的最小化，避免了由于频繁建连造成的计算倾斜。

#### Card 8. 连接迁移 (Connection Migration) 与双向路径验证机制
*   **技术机制**: 当客户端从 WiFi 切换到 5G 导致其网络 IP 和端口物理变更时，QUIC 依旧使用原来的 Connection ID。H2O 接获新包后，立即发送一个带 64 位随机数的 `PATH_CHALLENGE` 帧到客户端的新 IP，待客户端回应相同的 `PATH_RESPONSE` 帧后，才验证客户端的合法性并更新路由地址，防止反射式 DoS 攻击。
*   **应用策略**: 用于移动端接入加速，确保在电梯、室外等复杂切换场景下下载不断线、视频不卡顿。

#### Card 9. 自研 AES-GCM Fusion 融合硬件加速引擎的零拷贝发包流程
*   **技术机制**: QUIC 要求对每个输出的 UDP 数据包执行包头与包体双重加密。H2O 集成的 `picotls` 中自带了 `fusion` 加密引擎，利用 AVX2/AVX512 寄存器，将 AES-NI 加密与 Ghash 的乘法计算指令级融合在一个执行链内，使大块内存切片在单 CPU 周期内并行完成加密和数据拷贝。
*   **应用策略**: 在 x86_64 服务器上建议启用支持 AVX512 的编译标志，以将 HTTP/3 阶段的整体加解密算力开销减半。

## 📂 M3: HTTP/3 协议栈与 QPACK 头部压缩机理 (Cards 10-13)

#### Card 10. h2o_http3_conn_t 上下文结构与单向控制流状态机处理
*   **技术机制**: `h2o_http3_conn_t` 结构体是 L7 协议与 `quicly` 传输层之间的桥梁。它在建连初期自动启动一个专用的单向控制流（Control Stream），用于在双端之间宣告 `SETTINGS` 帧（包括并发流上限、QPACK 表大小），并使用控制流处理协议级异常与流状态重置。
*   **应用策略**: 确保两端 `SETTINGS` 参数一致，避免因配置差异引发意外的连接重置。

#### Card 11. QPACK 编解码器的表项管理与索引映射
*   **技术机制**: QPACK 是 HTTP/3 头部压缩规范。H2O 维护一个 99 项的静态表，以及一个在内存中动态分配、最大长度可配的动态表（Dynamic Table）。动态表缓存频繁出现的自定义报头（如 User-Agent、Cookie），编解码时用 1 字节索引代替冗长的字符串，实现报头的极致压缩。
*   **应用策略**: 调优 `hpack-capacity` 参数，在内存开销和压缩效率之间取得最佳折衷。

#### Card 12. QPACK 编解码单向同步流 (Encoder/Decoder Stream) 阻断排空
*   **技术机制**: 动态表项的增删和引用必须双端完全对齐。为此，H2O 开辟了两个专用单向流：Encoder Stream 和 Decoder Stream。当编码端在动态表添加新项时，必须通过 Encoder Stream 告知对端；对端解码成功后发送确认。若在未收到确认前发送了依赖新项的请求，该流将阻断解析，直至确认到达以防解码崩溃。
*   **应用策略**: 合理限制并发未决请求数，避免在高延迟网络下发生由于单向同步引起的流死锁。

#### Card 13. HTTP/3 多流无序传输与 QPACK 延迟解析的抗 HOLB 保障
*   **技术机制**: 得益于 QUIC 的 UDP 特性，HTTP/3 底层各个 stream 的 packet 在传输网络中允许乱序到达。H2O 对每个 Stream 独立进行 Frame 帧构建和解码。若某个流的 QPACK 动态表数据尚未完全就绪，H2O 会将该流的解码任务延迟挂起，其余流的业务解析和执行完全正常，实现了真正的流间隔离。
*   **应用策略**: 极其适合边缘网络、基站切换等易丢包场景，使得重传局限在具体流中，全局网关依旧保持高吞吐。

# Page 2 (H2 调度、零拷贝与安全容灾)

## 📂 M4: HTTP/2 依赖优先级调度树 (Cards 14-18)

#### Card 14. h2o_http2_scheduler_node_t 调度树的依赖拓扑与权值继承
*   **技术机制**: H2O 在 `lib/http2/scheduler.c` 中维护了一个树状优先调度结构。每个活跃的 HTTP/2 逻辑流都被映射为一个 `h2o_http2_scheduler_node_t` 节点。节点之间通过 Parent-Child 指针关联。每个节点持有特定的绝对权重，子节点按照权重比例瓜分父节点所分配到的物理带宽额度。
*   **应用策略**: 浏览器可以通过 `PRIORITY` 帧动态修改树的结构，以调整核心渲染资源的优先级。

#### Card 15. h2o_http2_scheduler_run 调度器加权赤字轮询 (WDRR) 分配算法
*   **技术机制**: 调度树的遍历如果采用常规深度优先搜索，计算复杂度会随着并发流数暴增。H2O 采用加权赤字轮询（WDRR）算法：为每个节点维护一个赤字计数器（Deficit）。调度运行 `h2o_http2_scheduler_run` 时，快速扫描当前活动分枝，若节点赤字大于 0 则分发发送字节配额，否则增加赤字计数器并轮转，计算复杂度达到 $O(1)$。
*   **应用策略**: 此算法解决了超高并发（如单连接上万并发流）下的 CPU 性能衰退问题，确保调度器的高效运行。

#### Card 16. 16KB 数据分片发送限制与 TCP 发送缓冲区动态混流平衡
*   **技术机制**: 传统的服务器在发送响应时，会将几百 KB 的文件一次性塞给 Socket 缓冲区，导致排在后面的小包无法发出。H2O 调度器在执行底层写操作时，强制设置 16KB 发送配额上限。当流发送完 16KB 的数据块后，系统立即重新执行 WDRR 优先级树轮询，重新分配带宽，保证关键 CSS/JS 响应不受阻塞。
*   **应用策略**: 通过 16KB 分片粒度的混流控制，在不影响大文件传输吞吐的前提下，最大化小文件的响应实时性。

#### Card 17. 服务端启发式权重调度器策略与主流浏览器缺陷消解
*   **技术机制**: 部分浏览器（如早期 Chrome）未能正确实现 HTTP/2 依赖优先级规范，甚至将所有流挂载为根节点的同级平铺节点，导致优先级调度形同虚设。H2O 内部集成启发式分析引擎，在检测到这种平铺结构后，自动干预并改写权重分配，提高 CSS/JS 文件的权重，保证浏览器客户端的渲染耗时最小。
*   **应用策略**: 属于开箱即用的协议优化，无需在前端编写任何适配逻辑即可提速页面加载。

#### Card 18. HTTP/2 流并发上限控制与 WINDOW_UPDATE 流量滑动窗口管理
*   **技术机制**: H2O 严格遵守 `SETTINGS_MAX_CONCURRENT_STREAMS` 限制（默认 100）。连接双端都维持着发送 and 接收的 `WINDOW_UPDATE` 滑动窗口。当客户端流耗尽发送窗口时，数据写入被自动挂起，直到收到对端返回的窗口更新帧，避免发送端缓冲区爆仓。
*   **应用策略**: 优化滑动窗口大小可以解决在高带宽延迟积（BDP）的长途跨国网络中因流控太小导致的限速。

## 📂 M5: 零拷贝内存管理与 I/O 缓冲区优化 (Cards 19-23)

#### Card 19. 轻量级 I/O 向量 h2o_iovec_t 的结构与高效指针切片
*   **技术机制**: H2O 内部通过 `h2o_iovec_t` 表示内存片段。它只包含一个字符型 `base` 指针与一个 `len` 长度。该结构是非拥有型（Non-owning）的切片，在数据封装、多层代理转发、报头截取时，直接通过指针偏移表示新切片，绝对不产生任何数据拷贝，极度节省内存。
*   **应用策略**: 在编写 C 自定义扩展模块时，应当遵循 `h2o_iovec_t` 生命周期原则，避免在异步回调中访问已释放的栈内存。

#### Card 20. h2o_buffer_t 双向链表动态内存池分配与释放回收
*   **技术机制**: 接收请求载荷时，H2O 使用 `h2o_buffer_t` 动态扩展双向链表。对于小于阈值（通常为 64KB）的数据直接从 Worker 独占的内存池（Mempool）中分配高速缓冲区；一旦数据超出阈值，自动转换为临时文件存储，既保证了小包处理的高性能，又防止了大请求体打爆网关内存。
*   **应用策略**: 生产环境中可调整 `max-request-entity-size`，以应对极高并发下的大包突发场景。

#### Card 21. TCP sendfile 内核零拷贝发送实现
*   **技术机制**: H2O 在处理静态大文件下载时，弃用常规的 `read` ➜ `write` 循环。在确定文件路由后，直接向事件循环底层的套接字写入文件描述符，底层通过系统 `sendfile()` 绕过用户态内存，直接在内核空间将文件 Page Cache 与 TCP 缓冲区进行零拷贝衔接。
*   **应用策略**: 配合操作系统级的 Page Cache 预读，可以使 H2O 的静态文件 serving 能力接近物理网卡理论极限。

#### Card 22. UDP GSO 批量发送与 sendmmsg 内核优化
*   **技术机制**: UDP 的单次发送开销远高于 TCP。为支撑 HTTP/3 高包率，H2O 支持 Generic Segmentation Offload (GSO)。当发送多个 QUIC 帧时，H2O 在内存中将它们打成一个大型包，利用 `sendmmsg()` 并携带套接字选项一次性投递给内核，由网卡物理硬件芯片在发送前执行物理切片，极地释放了 CPU 的中断计算开销。
*   **应用策略**: 在高带宽 HTTP/3 服务网关上，必须开启支持 GSO 的 Linux 内核版本（4.18+）以保证最高性能。

#### Card 23. 套接字写缓冲区回压 (Backpressure) 与异步可写事件挂起
*   **技术机制**: 当客户端接收慢或者物理链路发生严重拥塞时，H2O 写入 Socket 会返回 `EAGAIN` 错误。H2O 的 `h2o_socket_t` 立即停止后续数据生成，将未发送的 `h2o_iovec_t` 切片链挂起在内存发送队列中，并在 `evloop` 中注销写事件。等到内核通知 Socket 可写后，才重新触发回调发包，形成完美的回压流控。
*   **应用策略**: 回压机制从物理上阻断了慢速客户端耗尽服务器网关内存的隐患，保证了系统多租户隔离的公平度。

## 📂 M6: 高级特性与容灾优化 (Cards 24-28)

#### Card 24. 103 Early Hints 规范的流式发送与静态资源抢跑推送
*   **技术机制**: 客户端发起请求后，H2O 可以根据配置，在后台 CGI/反向代理尚未计算出 200 OK 响应前，立即先向客户端返回包含 Link 头的 103 Early Hints 状态码响应。客户端根据 Link 头，在等待主响应期间提前并行去 DNS 解析和下载关联的 JS 和 CSS 资源。
*   **应用策略**: 适用于复杂的现代 Web SPA 应用，配合 H2O 自身的 HTTP/2/3 多路复用，优化页面整体性能。

#### Card 25. TLS Session Ticket 会话状态的外部存储热加载与分布式同步
*   **技术机制**: TLS 会话 Ticket 中包含加密的会话状态。为了在 H2O 多实例集群中实现完全的 0-RTT/1-RTT 会话复用，H2O 支持配置外部会话 Ticket 密钥文件并自动进行 inotify 监听热加载。当密钥文件通过脚本动态更新时，多台 H2O 会自动重新读取内存密钥，实现全集群任意节点对同一客户端 Ticket 的无感解密。
*   **应用策略**: 运维团队可通过 Cron 任务定期滚动生成并分发 Ticket 密钥文件，兼顾集群连接复用率和前向安全性。

#### Card 26. Quicly 中拥塞控制算法的可插拔接口与 Cubic 状态转移
*   **技术机制**: `quicly` 抽象出可插拔的拥塞控制层。内置实现 `cubic` 与 `reno` 两种拥塞策略。连接初始化时在上下文注册 `quicly_cc_type_t` 结构。每次收到 ACK 或检测到丢包，`quicly` 都会执行该结构中的 `on_acked` 或 `on_lost`，并在 CUBIC 状态机中转移参数（如乘性减小、加性增大），精确估算最佳带宽延迟积（BDP）。
*   **应用策略**: 可以根据所处局域网、城域网或公网的环境特点，编译配置更适合当前带宽特征的拥塞控制插件。

#### Card 27. 优雅停机连接 Draining 与流平滑排空
*   **技术机制**: 收到系统 TERM 退出信号后，H2O 首先注销全局监听端口，拒绝接收任何新建连接。对于现存 TCP 连接，对于 HTTP/2 发送携带最大流 ID 的 `GOAWAY` 帧，对于 HTTP/3 触发 QUIC 的 Connection Draining。通知客户端当前连接即将断开，使现有事务在预设优雅超时内平滑执行完毕后，再平稳退役进程。
*   **应用策略**: 极大减少了生产发布和版本更新时的请求中断故障，实现无缝不停服滚动更新。

#### Card 28. 抗 HTTP/2 & HTTP/3 慢速泛洪攻击流控防护策略
*   **技术机制**: 恶意客户端可能发起大量空逻辑流（如发完 HEADERS 帧后不再发送后续，或以极慢速率发包）企图占满服务器的流通道句柄。H2O 实施了严格的重置流频率计数（RST_STREAM rate-limiting）与并发流超时约束。一旦连接累计重置流次数超限或单个 HEADERS 占位时间过长，H2O 会立即关闭此物理连接，确保正常请求的转发资源。
*   **应用策略**: 在互联网公开网关上，合理配置 `max-concurrent-streams` 和相关的 DoS 防御计时器，保障公网接入的高可用。

## ⚔️ H2O 核心运行参数与系统级隔离命令字典 (Page 2 Bottom)

### T1: H2O 核心运行与调优配置参数
*   **`num-threads: <integer>`**：配置 H2O 工作线程数，生产环境通常等于物理 CPU 核心数以消除核心间锁竞争。
*   **`hpack-capacity: <bytes>`**：QPACK/HPACK 动态表大小限制，默认 4096 字节。调大可提升长链接重复头部压缩比。
*   **`early-hints: ON | OFF`**：配置是否开启 103 Early Hints 抢跑资源推送，提前触发客户端下载关联 CSS/JS 静态文件。
*   **`limit-request-body: <bytes>`**：限制最大请求体尺寸，防止大体积恶意上传耗尽服务器工作内存和临时磁盘。
*   **`http2-idle-timeout: <seconds>`**：HTTP/2 连接空闲超时时间，超时后优雅断连以回收未决句柄及流控制内存占用。

### T2: 系统级隔离、调试与客户端诊断指令
*   设置系统套接字最大发送和接收缓冲区以防高包率 UDP 溢出丢包：
    `sysctl -w net.core.wmem_max=16777216`
    `sysctl -w net.core.rmem_max=16777216`
*   提高进程文件句柄上限以承载百万级高并发客户端连接：
    `ulimit -n 1048576`
*   使用 curl 客户端发起 HTTP/3 请求进行首包与控制流校验：
    `curl --http3 https://127.0.0.1:443/`
*   利用 OpenSSL 诊断 HTTP/3 TLS 1.3 握手与 ALPN 协商详情：
    `openssl s_client -connect 127.0.0.1:443 -alpn h3`
