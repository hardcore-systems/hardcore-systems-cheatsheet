# jepsen-io / maelstrom (分布式系统一致性与容错测试床)

## M1: Clojure 运行层与内核拓扑

### Card 1: Clojure JVM 运行时
- **核心机制**：Maelstrom 底层由 Clojure 编写，运行在 JVM 之上。它作为一个进程运行测试环境，通过启动多个子进程（待测节点）并控制它们之间的虚拟网络。
- **内存折衷**：JVM 的 JIT 编译器在运行期优化 Clojure 代码，但启动开销大且内存占用高（默认需分配至少 2GB 堆内存）。
- **命令行参数**：`-Xmx4g` 限制堆最大内存，`-server` 启用服务器模式 JIT。

### Card 2: STDIN/STDOUT 文本协议
- **交互规范**：Maelstrom 与待测节点之间通过标准输入输出（stdin/stdout）进行进程间通信。消息以单行 JSON 格式传输，必须使用 `\n` 换行符隔离。
- **死锁隐患**：节点代码中严禁写入多余的普通 stdout 日志（如 `println`），这会破坏 JSON 协议格式，导致 Maelstrom 无法解析而挂起。一切调试日志必须输出到 stderr。
- **诊断手段**：监控节点进程的 stderr 输出以抓取崩溃堆栈，利用 `tail -f store/latest/node.log` 查看。

### Card 3: 节点间消息通信
- **虚拟网络**：Maelstrom 会拦截并重定向节点间发送的消息。虽然节点感觉是在进行 Socket 通信，但本质上是 Maelstrom 在内存中进行包路由。
- **消息结构**：包含 `src`（源节点）、`dest`（目标节点）及 `body` 字典。`body` 中必含 `type` 和 `msg_id`。
- **协议折衷**：零真实网络 IO 开销，便于单机仿真数百节点，但无法测试真实的 OS 协议栈（如 TCP 握手重传等）缺陷。

### Card 4: 状态循环与并发
- **并发模型**：Clojure 的 Agent、Ref 与 Atom 构成内核的状态共享模型。Maelstrom 内部使用多线程处理来自不同节点的标准 I/O 字节流。
- **状态同步**：利用 Atom 进行轻量级无锁 CAS 状态累加，利用 STM（软件事务内存）确保集群拓扑的原子更新。
- **系统限制**：单机仿真并发量受限于物理 CPU 核数和 JVM 线程上限，建议单机测试节点数不超过 20 个。

### Card 5: JSON Schema RPC
- **协议格式**：RPC 请求包含 `type`（如 `init`、`txn`）、`msg_id`。响应消息需包含 `in_reply_to`，映射到对应的 `msg_id`。
- **异常定义**：如果发生超时或路由失败，响应 body 必须包含 `error_code`（10: Not Supported, 14: Txn Conflict）。
- **设计折衷**：JSON 文本编解码极其消耗 CPU 资源，但极易与各种主流语言（Python, Go, Rust, Java）编写的测试节点集成。

## M2: Jepsen 核心引擎与测试框架

### Card 6: Knossos 线性化校验
- **WGL 算法**：Knossos 利用 WGL (Wing & Gong) 算法对历史执行记录进行线性一致性判定，寻找一条合法的全局全序执行路径。
- **路径爆炸**：随着并发请求数和冲突率上升，线性化校验的搜索空间呈指数级爆炸。
- **优化手段**：设置 `--time-limit` 限制最长校验时间，避免内存溢出（OOM），对非交叉并发区间进行分段裁剪。

### Card 7: Elle 事务一致性分析
- **依赖图谱**：Elle 针对多 key 事务生成执行依赖图（包含写-写 WW、写-读 WR、读-写 RW 依赖关系）。
- **环路检测**：通过检测依赖图是否存在环路来寻找一致性违背（如 G1a：脏写；G-single：只读异常）。
- **技术优势**：相比 Knossos 仅支持单 key，Elle 能够在大规模、多 key 的复杂隔离级别（如 Serializable）下工作。

### Card 8: Generator 事件生成器
- **工作流管线**：Generator 定义了测试期间客户端的操作生成逻辑（例如随机生成读写、向指定 key 发起请求）。
- **并发控制**：利用线程池控制并行的客户端数。Generator 会以特定的 QPS 分发事件。
- **定制折衷**：可设计特定访问模式（如热点 key 争抢）来压测冲突处理，但可能偏离真实的随机生产环境负载。

### Card 9: Checker 归纳引擎
- **分析流**：测试执行完毕后，Maelstrom 会将所有的历史事件序列（History）保存到本地磁盘，并唤醒 Checker。
- **校验产物**：Checker 会生成 `results.edn` 文件，包含详细的一致性校验统计、通过/失败结论、以及反例执行序列。
- **分析指令**：利用 Clojure 脚本手动解析历史文件：`clojure -m jepsen.store`。

### Card 10: 客户端代理物理并发
- **物理映射**：每个客户端代理在 Maelstrom 内部绑定到一个独立的物理并发线程。
- **超时拦截**：默认 RPC 超时设置为 5000ms。超时未收到响应将被记录为 `info`（不确定状态），随后在一致性校验中判定是否允许其生效。
- **性能折衷**：增加线程数可以提升压测并发度，但也增加了 JVM 上下文切换的开销，影响时延精度。

## M3: 一致性模型与定理约束

### Card 11: 因果一致性验证
- **偏序关系**：因果一致性（Causal Consistency）要求有因果关联的写操作必须在所有节点以相同顺序展现。无因果关联的操作可以并发乱序。
- **向量时钟**：实现上通常在消息体中携带向量时钟（Vector Clock）或 Lamport 时间戳。
- **读写折衷**：读写性能高且无需跨节点锁同步，但在面临网络分区时无法保证全局最新的读取。

### Card 12: 读未提交与已提交
- **隔离级区别**：
  - **Read Uncommitted**：允许事务读取到其他未提交事务所写的临时脏数据（Dirty Read）。
  - **Read Committed**：确保只能读取到已提交的数据，避免脏读。
- **测试诊断**：Elle 通过寻找 $G1a$ (脏写) 和 $G1b$ (脏读) 环路来检测违背。
- **实现成本**：读已提交需要写锁或在 MVCC 中依据当前活动事务列表（Active Txn List）做可见性判定。

### Card 13: 可重复读
- **读取约束**：在同一事务中，多次读取同一 key 必须返回相同的值，不允许不可重复读（Non-repeatable Read）。
- **幻读挑战**：由于只拦截了 key 的显式读写，Maelstrom 对幻读（Phantom Read）的检测需要依靠多 key 范围关联。
- **物理成本**：事务在执行期间需要持有共享锁，或者利用 Snapshot Isolation (SI) 读取快照。

### Card 14: 串行化一致性
- **最高隔离级别**：Serializable 要求所有事务的执行结果与某种串行执行的结果完全一致。
- **环路特征**：在 Elle 依赖图中不能存在任何包含 RW, WR, WW 边的环（即 $G2$ 异常环）。
- **代价折衷**：在高并发写场景下，为了保证 Serializable，会导致极高的事务冲突率和大量的回滚中断（Abort）。

### Card 15: 单调读写约束
- **会话保证**：
  - **Monotonic Reads**：如果用户读取了值 V1，后续的读取绝不能读取到比 V1 更旧的值。
  - **Monotonic Writes**：确保同一用户的写操作顺序物理生效。
- **物理实现**：通常将用户会话绑定到特定的副本节点，或者在客户端维护本地读取过的最新时间戳上限。

## M4: 分布式共享存储与服务

### Card 16: Lin-KV 强一致存储
- **服务接口**：Maelstrom 提供了一个内置的强一致性 KV 存储组件 `lin-kv`。
- **工作机制**：在测试期间，节点可以通过 `lin-kv` 的 HTTP / JSON 接口来存储全局状态。`lin-kv` 本质上是一个运行在 Jepsen 共识层上的强一致性数据库。
- **使用折衷**：调用 `lin-kv` 极易构建无状态一致性服务，但它的高延迟会拖慢整体 TPS。

### Card 17: Seq-KV 顺序一致存储
- **模型特征**：`seq-kv` 保证顺序一致性（Sequential Consistency）。所有节点的写操作以全局相同的某种顺序生效，但不承诺实时性。
- **性能折衷**：比 `lin-kv` 拥有更低的写入延迟，但在网络分区期间读取可能会返回 stale 数据。
- **适用场景**：适合同步不要求毫秒级实时的元数据配置。

### Card 18: Lww-Register 冲突合并
- **冲突合并**：Last-Write-Wins Register 是一种强收敛的 CRDT。当多个并发写冲突时，以时间戳最大者（Latest）为准覆盖旧值。
- **物理时钟隐患**：由于依赖物理时钟，如果节点间存在时钟漂移（Clock Skew），会导致逻辑上较晚的写入被错误丢弃。
- **调优手段**：优先采用逻辑时钟或物理+逻辑时钟（HLC）来消除对物理时钟精度的强依赖。

### Card 19: PN-Counter 状态合并
- **计数器 CRDT**：Positive-Negative Counter 允许节点在本地无锁地执行增加（P）或减少（N）操作。
- **合并算法**：定期在节点间广播本地的 P/N 数组。合并时，对各节点的 P 和 N 取逐元素最大值（`max(P[i], P_remote[i])`）。
- **设计优势**：网络分区恢复后，节点状态可以无缝自动收敛，完全避免了分布式锁的复杂开销。

### Card 20: 副本日志同步
- **复制模式**：节点实现分布式服务时，通常需要自主实现日志同步。包含同步复制（强一致，响应慢）与异步复制（最终一致，响应快）。
- **脑裂防范**：在写入日志前必须获得法定人数（Quorum: `N/2 + 1`）节点的确认，避免网络分区时两边同时接受写入。

## M5: 网络混沌注入与状态转换

### Card 21: 网络分区混沌注入
- **故障模拟**：Nemesis 会周期性地将网络划分为两个或多个不相通的隔离分区（例如，5节点被分为 `{n1, n2}` 和 `{n3, n4, n5}`）。
- **运行日志**：在 Jepsen 日志中展现为 `:info :nemesis :start :partition`。
- **协议收敛**：考验共识协议是否能在多数派分区维持正常写入，且少数派分区能正确拦截并拒绝写入。

### Card 22: 丢包率动态模拟
- **包丢弃机制**：Maelstrom 内部网络模拟器可根据配置的概率丢弃节点间的消息。
- **配置参数**：`--drop-rate 0.1` 表示随机丢弃 10% 的网络包。
- **系统健壮性**：如果待测节点没有实现超时重传与确认机制，会导致丢包发生后系统直接死锁或数据永久丢失。

### Card 23: 时延膨胀与乱序
- **延迟抖动**：模拟网络拥塞导致的数据包排队。
- **配置参数**：`--latency 100` 注入基准延迟，`--delta 50` 注入时延抖动，导致消息包在到达时发生乱序。
- **设计挑战**：待测节点必须能处理过期旧消息的晚到，通常需要在消息中附带唯一的序列号。

### Card 24: Nemesis 故障引入器
- **控制中心**：Nemesis 是 Jepsen 的故障注入核心组件。
- **调度策略**：通过 Clojure 状态机定义故障的时间线（如每 10 秒发生一次网络分区，持续 5 秒后恢复）。
- **运行命令**：在 Maelstrom 中使用 `--nemesis partition` 启用分区，使用 `--nemesis-interval 10` 调整频率。

### Card 25: Recovery 恢复重连
- **路由重建**：当 Nemesis 撤销故障（例如网络恢复）时，Maelstrom 会物理重连网络路由，允许所有节点重新直接通信。
- **状态同步**：考验少数派节点重连后，是否能自动向 Leader 发起日志追平（Catch-up），清洗无效的未提交锁状态。

## M6: 诊断调试与测试调优

### Card 26: Gnuplot 吞吐时延图
- **绘图引擎**：测试完成后，Maelstrom 自动调用系统的 Gnuplot 引擎生成图表。
- **关键图表**：
  - `latency-raw.png`：散点图，横轴为测试时间，纵轴为每个请求的响应时延（以蓝色表示成功，红色表示失败）。
  - `throughput.png`：折线图，展示系统每秒的成功写入吞吐率。
- **诊断调优**：如果时延在网络分区期间出现高大红点，说明故障隔离及客户端重试避让机制有待调优。

### Card 27: History 历史日志追踪
- **历史记录文件**：存放在 `store/latest/history.txt` 中。
- **记录格式**：每行代表一个客户端操作，包含 `:process`（客户端ID）、`:type`（`:invoke` 发起、`:ok` 成功、`:fail` 失败）、`:f`（函数名）、`:value`。
- **调试诀窍**：若线性一致性校验报错，直接搜索 history.txt 中的值变化，查看哪个进程读到了未提交的旧值。

### Card 28: 命令行参数调优字典
- **测试执行命令**：
  `./maelstrom test -w txn-list --bin ./my_node --node-count 5 --time-limit 30 --rate 100 --nemesis partition`
- **核心调优参数**：
  - `--node-count`：设置测试集群节点规模。
  - `--rate`：每秒客户端请求并发率。
  - `--concurrency`：客户端物理并发线程数。
  - `--key-count`：限定操作的 key 范围，key 越少冲突率越高。
